# TM-83 — concurrent vs sequential /api/generate

## TL;DR

- 10 concurrent POSTs to `/api/generate` succeed (200) end-to-end on a freshly-booted dev server (PORT=3083, gpt-4o-mini, single dev user, PRO tier).
- p50 / p90 / p99 wall latency:
  - **concurrent-10**: 4936 / 5122 / 5891 ms (avg 4996, max 5891)
  - **sequential-30**: 5379 / 6624 / 7445 ms (avg 5569, max 7445)
- Concurrent-vs-sequential **avg-latency speedup ≈ 1.11×** — i.e. concurrency helps very little; the bottleneck is the upstream OpenAI generation, not server CPU.
- Dev-server **RSS grew from ~350 MB → ~472 MB across 40 calls** (+122 MB total, +75 MB after the 10 concurrent burst alone). After cooldown the process did **not** release this memory in the test window. No crash, no leak unbounded — but worth tracking.
- **No 401, no 5xx, no Prisma deadlock observed.** All 40 calls succeeded.
- **Race condition discovered (TOCTOU on monthly quota)** — see "Findings → 1" below. This is a real bug; we filed it as a follow-up before completing this experiment.

## Method

Driver: `__tests__/perf/tm-83-concurrent.mjs` — Node 22 fetch only (axios was disallowed by the task spec; we did not add a new dep). Uses `Promise.all` for the burst, plain `for/await` for the sequential.

Auth: `/api/dev/auto-login` returns the `authjs.session-token` cookie which we reuse across all calls (one user, intentional — that's the realistic worst case for shared-state contention on `User.monthlyUsage`).

RSS: `lsof -iTCP:3083 -sTCP:LISTEN` → pid → `ps -o rss= -p $pid`. Sampled 3 times (before / after burst / after sequential).

Run command:

```sh
cd worktrees/TM-83-perf-concurrent
PORT=3083 npm run dev    # in another terminal
PORT=3083 node __tests__/perf/tm-83-concurrent.mjs
```

Raw results JSON: [`__tests__/perf/tm-83-results.json`](../../__tests__/perf/tm-83-results.json).

## Numbers

### Run on PRO tier (clean baseline)

| metric                      | concurrent-10 | sequential-30 |
| --------------------------- | ------------- | ------------- |
| ok / failed                 | 10 / 0        | 30 / 0        |
| wall total (ms)             | 5,893         | 167,061       |
| wall p50 (ms)               | 4,936         | 5,379         |
| wall p90 (ms)               | 5,122         | 6,624         |
| wall p99 (ms)               | 5,891         | 7,445         |
| wall avg (ms)               | 4,996         | 5,569         |
| wall max (ms)               | 5,891         | 7,445         |

| RSS sample            | KB      |
| --------------------- | ------- |
| before                | 350,576 |
| after concurrent (10) | 425,504 |
| after sequential (30) | 472,576 |
| **delta total**       | +122,000 KB (+119 MB) |

Throughput:
- concurrent: 10 req / 5.89 s = **1.70 req/s**
- sequential: 30 req / 167.06 s = **0.18 req/s**

The wall-clock advantage of concurrency is large (5.9s vs ~167s) but per-request latency only improves ~11%. That's expected: each request blocks on OpenAI's gpt-4o-mini, which dominates the wall budget. Server-side overhead (auth, prisma findUnique, prisma asset+version create, prisma user update) is not the bottleneck.

### Run on FREE tier (the eye-opener)

The first run was accidentally executed against the FREE-tier dev user (limit=3/month, usage=0):

- concurrent-10: **all 10 returned 200** (asset + DB row created) — the `monthlyUsage < 3` check passed for each in parallel before any of them committed `increment: 1`.
- sequential-30 immediately after: all 30 returned 429 because by then `monthlyUsage` had reached 10.

This is the TOCTOU race described in Findings → 1.

## Findings

### 1. Quota check is a TOCTOU race (real bug, file separate task)

`src/app/api/generate/route.ts`:

```ts
// L43: read
const limitCheck = checkGenerationLimit({ tier: user.tier, monthlyUsage: user.monthlyUsage });
if (!limitCheck.allowed) return 429;

// L55: 5–8s OpenAI call — by now, N other requests may have read the same value

// L98: write
await prisma.user.update({ data: { monthlyUsage: { increment: 1 } } });
```

10 concurrent requests for a FREE user with `monthlyUsage=0` (limit=3) all read 0, all proceed, all consume tokens, all commit. The free tier is overshot by 233% and the user pays nothing. On PRO (limit=200) the absolute risk is small, but the same race exists.

**Mitigation options** (for the follow-up task to choose):
- (a) Atomic conditional update: `prisma.user.updateMany({ where: { id, monthlyUsage: { lt: limit } }, data: { monthlyUsage: { increment: 1 } } })` BEFORE the OpenAI call; refund (decrement) on AiRefusal/error paths that already skip increment.
- (b) Per-user in-memory mutex (rejects 429 immediately if a request is in-flight) — cheap, but does not survive a crash mid-call.
- (c) Add a `pendingUsage` column and treat reservation+commit as two phases.

ADR-0001 says edit ≠ render, edit is LLM-only — so option (a) keeps the cost charge tied to the LLM commit point. Recommended.

### 2. Concurrency gives ~11% per-request speedup, ~28× wall-clock speedup

The per-request median moves from 5.38 s → 4.94 s when 10 are in-flight. That's noise-level for the user, but the total time to fan-out 10 requests drops from ~55 s to ~5.9 s — i.e. concurrent dispatch is dominated by the slowest individual call, not by server-side queueing. Conclusion: **the server is not the bottleneck at this scale; OpenAI is.** No need to add server-side queueing today.

### 3. Memory growth: +119 MB across 40 calls, no observed leak

RSS climbed 350 → 472 MB and did not retract within the test window. A few candidates:
- Turbopack dev mode keeps compiled module graphs hot.
- Prisma's query engine warms its statement cache.
- We did not collect heap snapshots; the growth fits "warming caches" more than "leak" — but we should re-run with `--inspect` and a heap snapshot if RSS continues to climb in production.

No requests failed with OOM, ECONNRESET, or 5xx. Acceptable for now; future task should re-measure on prod build (`next build && next start`) where Turbopack dev overhead is absent.

### 4. NextAuth + Prisma session lookup handled the burst cleanly

Every request authenticated against the same session token. Zero 401s. Auth-table contention is not a problem at N=10.

## Risks / unknowns

- We tested 1 user × 10 concurrent. We did **not** test N distinct users × 10 each (Prisma connection pool saturation behavior). Out of scope for TM-83 but a candidate follow-up.
- We tested on dev (Turbopack). Prod build numbers will differ. Out of scope.
- We used a short prompt; longer prompts (more tokens) shift the latency distribution. Acceptable for a baseline.
- Cost: ~40 OpenAI gpt-4o-mini calls × ~$0.05/call ≈ **$2** total spend. Within the requires_env budget.
