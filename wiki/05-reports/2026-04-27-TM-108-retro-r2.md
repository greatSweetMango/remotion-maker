# TM-108 r2 — Retro

## TL;DR

TM-111 머지 직후 TM-108 r2 재측정. **가용성 1/5 → 5/5 회복**(TM-111 의도대로), 그러나 클라이언트 evaluator 의 `SceneNParams` 식별자 미정의 문제로 multi-step 5/5 모두 ErrorBoundary 만 노출 — full 평균 25.2/100. baseline 은 70.6/100. 다음 회차(r3)는 새 fix 머지 후 자동 트리거.

## 잘된 점

- TM-111 의 sanitize 와 fallback 이 의도한 역할을 정확히 수행. **sanitize 1회 관측, fallback 0회** — sanitize 가 우선 방어선으로 충분.
- 5 케이스 100% generate-200, latency 도 r1 대비 소폭 단축 (22.5s → 19.0s).
- 실패 모드가 sandbox 가 아닌 evaluator runtime 으로 명확히 좁혀져, 다음 fix 의 작업 범위가 분리됨 (composer or evaluator).

## 아쉬운 점 / 다음에 고칠 것

- **품질이 baseline 보다 한참 낮음 (-45)** — 가용성만 보면 성공이지만 사용자 가시 결과는 회귀. multi-step 을 default-on 으로 가는 길은 여전히 멀다.
- 벤치 결과를 보고도 "200 OK" 만으로 회귀를 놓칠 수 있음 — judge 점수까지 들여다봐야 신호가 잡힌다. 차후 r3 부터는 **judge 평균이 baseline -10 이내** 같은 임계도 함께 측정 항목에 추가해야 함.
- evaluator 의 PascalCase 식별자 처리 (`SceneNParams`) 가 사전 검증되지 않은 채 multi-step 머지된 것이 근본 원인. 다음 multi-step 변경 시 **샘플 출력 → evaluator round-trip 단위 테스트** 가 PR 게이트가 되어야 함.

## 학습

- "가용성 회복" 과 "품질 회복" 은 별개 이벤트. TM-111 PR 머지 시점에 r2 재측정 박제를 묶어둔 것이 효과적이었음.
- gpt-4o 의 코드 출력 위생은 패턴이 추가될 때마다 sanitize 패스도 늘어야 함 — 모니터링 (`sanitize 발생 횟수 + 패턴 종류`) 을 telemetry 에 박제해야 sanitize 커버리지 평가 가능.
