# TM-69 회고 — OpenAI json_object 강제

## 무엇을 했나

TM-41 r2에서 보고된 invalid JSON 실패(ta-09)를 OpenAI `response_format: json_object` 옵션으로 봉쇄. 단위 테스트 4건 + 라이브 smoke 1건(5콜) 추가. 관련 system prompt 한 줄 보강.

## 잘 된 것

- 변경 범위가 작음 — client.ts 한 함수, 약 8줄 추가. 외부 시그니처 무변경.
- json mode 요건("json" 키워드)을 클라이언트 자체에서 보장하도록 자동 보강. 호출자가 system prompt를 바꿔도 안전.
- 라이브 5콜 5/5 PASS — 즉각적인 신호. 평균 응답 ~10s (gpt-4o-mini 정상 범위).
- 단위 테스트가 OpenAI 분기와 prompt 보강 양쪽을 다 커버 → 향후 회귀 방어.

## 아쉬운 것 / 학습

- TM-41 r2 보고서가 아직 작성되지 않은 상태에서 작업 — 원인 추정에 일부 의존했다. 다행히 기존 TM-41 보고와 OpenAI 문서 두 출처로 충분히 식별됨.
- 워크트리에 `node_modules`가 없어 메인 레포 jest 바이너리를 빌려 썼다. tsx도 미설치였다. 워크트리에서 ts 스크립트 직접 실행은 어렵고, jest 게이트(TM69=1) 패턴이 가장 마찰 적다 — 다음에도 같은 패턴 사용 권장.

## 다음에 더 잘 하려면

- OpenAI 외 provider 추가 시(예: Gemini, Mistral) "structured output" 옵션을 client.ts에서 통일된 인터페이스로 노출하는 ADR 후보. 지금은 OpenAI 1곳만 처리.
- 라이브 smoke 테스트는 cost 게이트(`TM69=1`)가 명시적이라 CI에 들어가지 않는다. 정기 nightly run에 포함시킬지는 별도 task로.

## 메트릭

- 코드 변경 LoC: +12 (client.ts +12, prompts.ts +1)
- 테스트 추가: +4 unit, +1 live (5 cases)
- 라이브 검증: 5/5 PASS, ~50s wall, < $0.01
- 작업 시간: ~25분
