# TM-113 회고

## TL;DR

Spec("NEXTAUTH_URL 자동 치환")과 실제 페인("재실행 시 .env.local 덮어쓰기")이 어긋난 케이스. 코드를 먼저 읽어 둘의 차이를 식별한 뒤, 진짜 페인을 해결하는 방향으로 좁혀 작업. ~10 LoC 변경 + 헤더 주석 정리. OpenAI 호출 0회. 추정 complexity=low 정확.

## 잘된 점

- Phase A에서 setup-worktree.sh 전체를 먼저 읽고, NEXTAUTH_URL 동적 치환이 이미 구현되어 있음을 확인 → 중복 작업 방지.
- 진짜 페인(재실행 시 cp가 사용자 변경 클로버) 식별 → 멱등성 의미를 "value-preserving"으로 재정의.
- 멱등성 검증을 실제 가짜 플래그(`TM113_TEST_FLAG=...`) 추가 후 재실행으로 수행 — 코드 리뷰만으로 통과시키지 않음.
- port 9999 → 3113 round-trip 으로 awk 치환의 양방향 동작 확인.

## 아쉬운 점

- spec 텍스트만 보고 바로 수정에 들어갔으면 헛수고할 뻔. 차후 비슷한 dev-tools 작업은 항상 "기존 코드 동작 확인부터" 원칙 명문화 가치 있음.
- retro/report에 직전 TM-112 retro 정확한 파일명 링크를 포함하지 못함(파일 식별 시간 절약). 사후에 fixup 가능.
