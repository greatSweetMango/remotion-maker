# TM-82 — Network resilience QA

## Scope

Verify the system's behavior on three transient failure modes for the
LLM dependency (`/api/generate`, `/api/edit`):

1. Provider returns `5xx` (e.g. 503 Service Unavailable, 529 Overloaded).
2. Request times out / aborts before any tokens stream.
3. Stream disconnects mid-flight after partial tokens have been delivered.

Acceptance:
- Quota is **never burned** for upstream failures (refund on 5xx / timeout).
- Retry is **idempotent** with respect to billing.
- User sees a clear error message AND a way to retry without re-typing.
- Partial / corrupted streams are NOT persisted as successful assets.

## Method

Failure injection via Jest mocks at the SDK layer (`openai`,
`@anthropic-ai/sdk`) — no live network. Three new test files:

- `__tests__/lib/ai/client-resilience.test.ts` — 6 tests, both providers.
  Covers 5xx propagation, timeout/abort, and mid-stream disconnect after
  partial deltas.
- `__tests__/api/generate/upstream-5xx-refund.test.ts` — 3 tests.
  Reserve-then-refund flow at the route boundary; double-retry asserts
  net-zero quota change after two successive 5xx attempts.
- `__tests__/components/prompt-panel-retry.test.tsx` — 5 tests.
  RTL-driven verification of the new error banner + Retry affordance,
  including disabled-during-loading guard and dismiss flow.

All 14 new tests pass (`npx jest __tests__/.../...resilience...` etc).

## Findings

### F1 — `chatCompleteStream` correctly propagates upstream errors (PASS)

The SDK creation calls (`client.chat.completions.create`,
`client.messages.stream`) re-throw provider errors with status + message
intact. No silent fallback / empty resolve path was discovered. Verified
for both providers.

### F2 — Mid-stream errors deliver partial deltas BEFORE rejecting (PASS, with caveat)

When a stream disconnects after some tokens have streamed, the
`onDelta` callback is invoked for the partials that did arrive, then
the awaited promise rejects. Caller contract: **never persist on
streamed text alone — only on a resolved promise**. Both `/api/generate`
and `/api/edit` already follow this (assets are created only after
`generateAsset` / `editAsset` resolve). No bug found.

Caveat documented at `client.ts` boundary so future streaming UI
overlays don't accidentally commit provisional text.

### F3 — Quota refund on 5xx works (PASS)

`POST /api/generate` reserves a slot via the TM-92 atomic `updateMany`,
then issues a single `decrement: 1` from the catch branch on any
exception. Verified with two consecutive failed attempts — net usage
delta is 0, refunds match reservations 1:1. Same pattern in `/api/edit`
on the editUsage map.

### F4 — User-facing affordance MISSING → fixed in this PR

Pre-existing UX gap: on 5xx, only a sonner toast was shown (auto-
dismisses ~5s) and the typed prompt was wiped from the textarea. The
user had to re-type their prompt to retry, which is a poor recovery
flow especially on long generation prompts.

**Fix shipped in this PR**:
- `useStudio` now tracks `lastFailed: { kind, prompt, answers? }` on
  every failed `generate` / `edit`. Cleared on success or on explicit
  dismiss. New hook actions: `retry()`, `dismissError()`.
- `PromptPanel` renders a persistent inline error banner
  (`role="alert"`, `aria-live="polite"`) with a "다시 시도" Retry button
  and a dismiss (X) button. Banner survives toast auto-dismiss; Retry
  is disabled while a request is in flight to prevent double-fire.
- Wired into both desktop and mobile call sites of `Studio.tsx`.
- Quota refund makes Retry idempotent with respect to billing —
  documented in inline comments at the catch branches.

### F5 — Existing client.ts has no explicit retry / timeout / backoff (deferred, not blocker)

`chatCompleteStream` does NOT wrap the SDK call in an explicit retry-
with-backoff or `AbortController` timeout. The Anthropic + OpenAI SDKs
both have built-in retry behavior (default `maxRetries: 2`, exponential
backoff) and an `httpAgent` timeout, so single-call resilience comes
"for free" today. We chose **not** to add an outer retry loop here:

- Adding our own retry would compound with SDK retries (4–6× cost
  amplification on a sustained 5xx outage).
- Manual retry via the new banner + button gives the user explicit
  control, which is preferable for an interactive workflow where the
  next prompt may differ.

Documented as a deferred consideration. If a future operator-side
incident shows real-user impact, a deliberate single retry on
network-class errors only (not on 4xx / refusals) could be added —
but that would be a new task with explicit observability.

## Reproduction snippets

```ts
// 5xx propagation
openaiCreate.mockRejectedValue(new APIError(503, 'Service Unavailable'));
await expect(chatCompleteStream({...})).rejects.toMatchObject({ status: 503 });

// Mid-stream drop with partial onDelta
openaiCreate.mockResolvedValue(throwAfter([{ choices:[{ delta:{ content:'{"title":"par' }}]}], new Error('drop')));
const seen = []; await expect(chatCompleteStream({..., onDelta: c => seen.push(c)})).rejects.toThrow();
expect(seen.join('')).toBe('{"title":"par'); // partial reached UI; promise rejected
```
