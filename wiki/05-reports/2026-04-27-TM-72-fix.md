# TM-72 — capture-side determinism fix

## TL;DR

- TM-46 r5 RCA pinpointed **capture-side LLM code-generation variance** as the dominant cause of the r4 → r5 acceptance-gate noise (judge-side variance was already removed by TM-70).
- Mirrored the TM-70 judge config on the **generation path** (`src/lib/ai/client.ts`): `temperature=0` + `seed=42` by default, env-overridable (`AI_TEMPERATURE`, `AI_SEED`).
- Live verification (5 prompts × 3 calls, gpt-4o-mini): **paramsΔmax dropped 49 → 7 chars (−86%)**, **codeΔmax mean 615 → 312 (−49%)**. 1/5 prompts is now byte-identical across all repeats; 2/5 collapse to 2 unique outputs out of 3.

## 무엇이 바뀌었나

1. **`src/lib/ai/client.ts`**
   - Added `DEFAULT_AI_TEMPERATURE = 0`, `DEFAULT_AI_SEED = 42`, `resolveTemperature()`, `resolveSeed()`.
   - OpenAI `chat.completions.create({ temperature, seed, … })` — both fields now sent on every call.
   - Anthropic `messages.stream({ temperature, … })` — `temperature` pinned (Anthropic SDK does not accept `seed` yet, so omitted).
   - Env knobs: `AI_TEMPERATURE` (float, default `0`), `AI_SEED` (int, default `42`; `none` to omit entirely for A/B variance studies).
2. **`__tests__/lib/ai/client-stream.test.ts`** — 4 new tests
   - Default OpenAI request includes `temperature: 0, seed: 42`.
   - Default Anthropic request includes `temperature: 0`, never `seed`.
   - `AI_TEMPERATURE=0.7 AI_SEED=7` overrides forwarded as expected.
   - `AI_SEED=none` omits the field entirely (request must NOT include `seed`).
3. **`scripts/qa/tm-72-capture-variance.mjs`** — live A/B harness (5 prompts × 3 repeats × 2 conditions = 30 OpenAI calls).
4. **`__tests__/benchmarks/results/tm-72/variance.json`** — full pairwise edit-distance + hash matrix from the live run (committed for audit).

## 왜 / 배경

TM-70 fixed judge-side variance (gpt-4o vision rubric) by pinning `temperature=0, seed=42`. TM-46 r5 then re-judged the same r4 PNG set with the new judge config and still saw a delta vs r3 — RCA pointed at the **capture step** (the LLM that emits Remotion TSX) being non-deterministic: identical prompts produced different particle counts, easing curves, and color palettes between runs, so each r-N capture batch was effectively a new sample drawn from a wide distribution. This invalidates per-iteration fix/regression attribution.

The fix is the smallest possible change that mirrors what already worked for the judge: pin `temperature` to 0 and pass an explicit `seed`.

## 영향

### 측정 결과 (live, gpt-4o-mini, 2026-04-27)

5 prompts × 3 repeats × 2 conditions = **30 OpenAI calls**.

#### Per-prompt summary

| prompt | uniqueFullHashes A → B | codeΔmax A → B | paramsΔmax A → B |
|---|---|---|---|
| P1 typewriter "HELLO WORLD" | 3/3 → **1/3** | 392 → **0** | 11 → **0** |
| P2 8 bouncing circles | 3/3 → 3/3 | 969 → 1210 | 132 → 27 |
| P3 progress bar 0→100% | 3/3 → **2/3** | 548 → **12** | 8 → **0** |
| P4 pulsing star scaling | 3/3 → **2/3** | 635 → 321 | 34 → **0** |
| P5 three-line fade-in | 3/3 → 3/3 | 529 → 18 | 61 → 8 |

#### Aggregate

| metric | A (default) | B (temp=0, seed=42) | Δ |
|---|---|---|---|
| mean codeΔmax (Levenshtein, chars) | 615 | **312** | **−49%** |
| max codeΔmax | 969 | 1210 | +25% (driven by P2 outlier) |
| mean paramsΔmax | 49 | **7** | **−86%** |
| max paramsΔmax | 132 | **27** | **−80%** |
| prompts with all-unique hashes | 5/5 | **2/5** | −60% |

### 해석

- **PARAMS block** (the structured part the customize UI binds to per ADR-0002) is now **near-deterministic** — paramsΔmax mean dropped from 49 chars to 7. This is the field that drives the visual judge rubric most directly (particle count, duration, color, easing). std≈0 for 4 of 5 prompts.
- **Full code body** is partially deterministic. P1 byte-identical across all 3 calls. P3/P4 collapse from 3 unique outputs to 2 (one drift run out of three). P2 + P5 still vary — consistent with OpenAI's published statement that `seed` is **best-effort**, not guaranteed, especially for longer free-form completions.
- **Net acceptance-gate effect** (estimated): per-sample Δmax noise from the capture path should drop from the order of the prior judge variance (~10 score points) to single-digit, since the judge sees PNGs whose underlying `PARAMS` are now stable. End-to-end r5/r6 noise band can be tightened to roughly ±5.

### 사용자/제품 영향

- 같은 prompt 를 두 번 입력하면 *훨씬 더 비슷한* 영상이 나옴 (특히 PARAMS 가 동일). UX-wise 이는 "AI 가 들쭉날쭉하지 않다" 인상으로 긍정적.
- A/B 실험을 의도적으로 돌리고 싶을 때 `AI_SEED=none` 또는 `AI_TEMPERATURE=0.7` 로 opt-in 가능.

### 비용/성능

- API 호출 비용·지연 변화 없음 (parameter 추가만).
- prompt-caching 키 안정성 영향 없음 (system prompt 미변경).

## 한계 / 솔직한 회고

- `seed` 는 best-effort 임. P2 (8 bouncing circles) 같은 긴 free-form 코드는 여전히 회마다 다른 코드를 만든다 (codeΔmax 1210 chars). PARAMS 만 안정화되어도 acceptance-gate noise 는 충분히 줄지만, "동일 prompt = 동일 영상" 보장은 아님.
- 대안으로 (1) prompt 자체에 "respond with the same code if the prompt is identical" reinforcement, (2) cache layer at the prompt-hash level, (3) JSON-schema 강제로 코드 구조를 더 좁힘 — 이 셋 중 (2) 가 ROI 가 가장 큼. 별도 task 로 박제 권장.
