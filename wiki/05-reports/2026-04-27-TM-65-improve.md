# TM-65 — Multi-worktree cookie isolation

## Problem

Discovered during TM-43 template-visual-audit (see `wiki/05-reports/2026-04-27-TM-43-template-visual-audit.md`):
running two parallel dev servers from sibling git worktrees — both bound to `localhost`
on different ports (e.g. `:3043` and `:3056`) — caused a NextAuth login on one port to
silently bounce/redirect over to the other worktree's dev server. Sometimes the user
even ended up "logged in as a different test account" without doing anything.

## Root cause

Browsers scope cookies by **hostname**, not by `host:port`. RFC 6265 §5.1.3:

> A cookie's "request URI" is the URI of the HTTP request... The cookie's domain
> matches the request URI's host, NOT including the port.

NextAuth's default session cookie (`next-auth.session-token`) is set with no explicit
`domain` attribute, so the browser stores **one** cookie keyed by hostname `localhost`
and sends it on every request to any port on `localhost`. Worktree A's login
overwrites worktree B's session cookie. The resulting cookie may be valid against
worktree A's session secret but worktree B's middleware reads the same cookie value,
sometimes accepts it (same `NEXTAUTH_SECRET`), and routes the user accordingly.

Even when the secret differs, the cookie clobber alone is enough to log the user out
of the other worktree on every login — a constant context-switch tax.

## Fix

`scripts/setup-worktree.sh` (TM-56) now writes `NEXTAUTH_URL` using a per-worktree
hostname, defaulting to `127.0.0.1` instead of `localhost`. New signature:

```bash
bash scripts/setup-worktree.sh <worktree_path> <dev_port> [hostname]
```

- Default hostname: `127.0.0.1`. Browsers treat `127.0.0.1` and `localhost` as
  separate hosts → separate cookie jars.
- Optional 3rd arg: any hostname matching `[A-Za-z0-9.-]+`, e.g. `tm-43.local`
  (mapped via `/etc/hosts`) for a third concurrent worktree.

Recommended assignment for daily multi-worktree work:

| Repo / worktree | Hostname |
|---|---|
| main repo | `localhost` |
| worktree A | `127.0.0.1` |
| worktree B+ | `tm-<id>.local` (add to `/etc/hosts: 127.0.0.1 tm-<id>.local`) |

Started the dev server with the matching hostname:

```bash
PORT=3065 HOSTNAME=127.0.0.1 npm run dev
# then visit http://127.0.0.1:3065
```

## Why not also touch NextAuth `cookies` config?

Considered scoping the cookie name per-port (e.g. `next-auth.session-token.3065`).
Rejected: brittle (every cookie NextAuth issues — CSRF, callback, PKCE — would need
the same suffix), and adds prod risk for a dev-only problem. Hostname separation is
the standards-aligned answer and requires zero application code changes.

## Verification

- `bash -n scripts/setup-worktree.sh` — syntax OK.
- Bad-hostname guard rejects `"bad host"` (space) with exit 64.
- Awk substitution smoke test confirmed `NEXTAUTH_URL=http://127.0.0.1:3065`
  written when default hostname is used.
- Existing worktrees keep working — they still have `localhost:<port>` in their
  `.env.local`. Re-running `setup-worktree.sh <path> <port>` migrates them to
  `127.0.0.1:<port>` and clears the conflict.
