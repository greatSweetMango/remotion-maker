# TM-102 r2 회고

## 한 줄 요약

TM-111 sanitize/fallback fix 후 TM-108 의 4 sandbox 실패 케이스를 라이브 재실행 →
**5/5 200, fallback 0, sanitize 1×**. 단순 검증이지만 결정적 신호.

## 잘된 점

- **재현 정확성**: 동일 5 프롬프트 + 동일 모델/플래그 → before/after 비교가 1:1 로 깔끔.
- **저비용 게이트**: judge/screenshot 생략으로 \$1.4 → \$0.5 절감. 가용성만 확인하면 되는 r2 의
  목적과 일치.
- **검증 드라이버 재사용성**: `tm-102-r2-verify.ts` 는 향후 sandbox 회귀 회피 smoke 로 재사용 가능
  (CI 에서 nightly 1 회 도는 형태로).

## 아쉬운 점

- **표본 5 만으로 default-on 결정 불가** — sanitize 가 6 패턴을 cover 하지만 새 변종은 측정 못 함.
- **judge 부재**: "200 떴다" 와 "유의미한 영상이 나왔다" 는 다르다. 본 r2 는 전자만 본다.
  TM-108 이 case 2 long-video 를 200 인데 judge 15 점 (빈 화면) 으로 잡았던 패턴이 다시
  나올 가능성 — 후속 judge run 필요.
- **dev log 의 `[TM-111] scene-code[1] auto-sanitized` 신호가 1 줄 뿐** — sanitize 적용 빈도/
  패턴별 카운터를 telemetry 로 export 하면 운영 가시성↑.

## 다음에 할 것

- (필수) 프로덕션 telemetry: `multiStep.fallback` 카운터 + sanitize 패턴별 hit 카운터.
- (권장) TM-108 judge 재실행: 본 r2 에서 생성된 5 asset 에 대해 동일 4축 루브릭 채점 → 품질 회복도
  정량화 (\$1.4).
- (선택) 표본 확장: 카테고리당 3~5 케이스로 늘려 sandbox 회귀 smoke 를 nightly 화.

- **단계별 fix 검증의 가치**: TM-108 (RCA) → TM-111 (fix) → TM-102 r2 (verify) 의 3-step 체이닝이

  컨텍스트로 직접 사용된다.
- **사용자 가시 warning 의 중요성**: 응답에 `multiStep.fallback` 플래그가 있으면 silent 회귀를 막을
  수 있다. 본 r2 에서는 0 회 발동했지만, telemetry 가 있다면 production 에서 이 신호를 보고 커버
  미스 패턴을 즉각 발견 가능.
