# TM-89 — /api/export bundle entry registers RemotionRoot

## TL;DR

- `/api/export` 가 bundle entry 로 `UniversalComposition.tsx` 를 직접 썼지만 그 모듈은 `registerRoot()` 를 호출하지 않아 standalone `@remotion/bundler` 경로에서 깨졌다.
- TM-75 의 `vr-entry.tsx` 패턴을 따라 `src/remotion/export-entry.tsx` 를 추가하고 라우트가 이 entry 를 가리키도록 변경 → 단위 테스트로 `selectComposition` 이 정상 resolve 됨을 확인.
- Bundler webpack alias 에 `@/*` 를 주입해야 함 (TM-75 도 동일 사유로 주입). 라우트와 테스트 양쪽에 적용.

## 무엇이 바뀌었나

- 신규: `src/remotion/export-entry.tsx` — `RemotionRoot` 를 import 해 `registerRoot()` 호출하는 thin entry.
- 변경: `src/app/api/export/route.ts` — `entryPoint` 를 새 entry 로 교체, `webpackOverride` 에 `@/*` alias 주입.
- 신규: `__tests__/api/export/bundle-entry.test.ts` — 실제로 bundle 후 `selectComposition('UniversalComposition')` 이 fps/width/height 를 반환함을 검증. `REMOTION_BUNDLE_TEST=1` 환경변수로 opt-in (Chrome Headless Shell 다운로드 + ~20s).

## 왜 / 배경

TM-75 가 visual-regression 드라이버를 만들면서 발견한 사실: `UniversalComposition.tsx` 는 `RemotionRoot` 를 export 하지만 `registerRoot()` 는 호출하지 않는다. In-app `<Player>` 경로는 자체 런타임이 implicit 하게 등록해 주지만, `@remotion/bundler` 의 standalone 경로는 명시적인 `registerRoot()` 가 필요하다 → 호출이 없으면 bundle 의 root registry 가 비어 `selectComposition` 이 throw 한다.

`/api/export/route.ts` 가 정확히 이 standalone bundler 경로를 사용하므로 export 자체가 동작하지 않는 상태였다. TM-75 가 vr-entry 패턴을 이미 검증해 두었기 때문에 동일한 패턴을 export 경로에도 적용했다.

## 영향

- 코드: `src/app/api/export/route.ts`, `src/remotion/export-entry.tsx`, `__tests__/api/export/bundle-entry.test.ts`.
- 사용자/제품: GIF/MP4/WebM export 가 실제로 동작 가능해짐 (이전엔 첫 호출에서 bundler 단계가 깨졌음). ADR-0001 (Edit ≠ Render) 와 충돌하지 않음 — 여전히 export 시점에만 렌더한다.
- 성능: 첫 export 호출이 webpack bundle (수 초) 을 트리거하지만 결과는 모듈-레벨 `bundleCache` 에 보관되어 후속 호출은 즉시.
