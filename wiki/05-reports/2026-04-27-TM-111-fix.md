# TM-111 — Pipeline sandbox fallback

## TL;DR

TM-108 벤치에서 multi-step 파이프라인(`AI_MULTI_STEP=1`, gpt-4o)이 5건 중 4건 sandbox/transpile 실패로 500 을 떠
가용성이 사실상 붕괴됐다. 본 패치는 (1) scene-code 단계와 composer 단계에 **forbidden-token 자동 정제** 패스를
추가하고, (2) 어떤 stage 든 sandbox/transpile 로 throw 시 **single-shot generate 로 자동 fallback** 하도록
`generateAssetMultiStepAsApiResponse` 를 감쌌다. 단위 테스트로 4 실패 패턴(`require`, `globalThis`, dynamic
import, `Function` ctor) 모두 sanitize 후 sandbox 통과 확인. fallback 경로는 dynamic import 로 generate.ts
를 호출하여 순환 의존을 회피한다.

## 무엇이 바뀌었나

- `src/lib/ai/pipeline.ts`
  - 신규 export `sanitizeForbiddenTokens(input)` — gpt-4o 가 자주 흘리는 6 종 패턴을 사후 제거/치환:
    - `const x = require('...')` → 줄 삭제
    - 잔존 `require(...)` → `undefined`
    - `globalThis.X` / `globalThis['X']` → 베어 식별자 `X`
    - 동적 `import(...)` → `undefined`
    - `new Function(...)` / `Function('...', '...')` → `(() => null)`
    - `process.X` → `undefined`, `import.meta` → `undefined`
  - `generateSceneCode` — `validateCode` 호출 **전에** 위 sanitizer 를 우선 적용. 여전히 invalid 면 새
    `SceneSandboxError` 를 throw 하여 호출부가 식별 가능하게 함.
  - `generateAssetMultiStep` — composer 산출물에도 동일 sanitizer 적용 후 validate. asset.code 도 scrubbed
    버전을 영구 보관하여 후속 edit/render 사이클이 깨끗한 소스를 본다.
  - `generateAssetMultiStepAsApiResponse` — try/catch 로 감싸고, pipeline-origin error(`TM-102 …`,
    `sandbox validation`, `failed to transpile`, `did not return valid JSON`) 또는
    `SceneSandboxError` 발생 시 `generateAsset` 단일-샷으로 fallback. 응답에
    `multiStep.fallback: 'single-shot'` 플래그 + `warning` 문자열을 추가하여 관측 가능.
- `__tests__/lib/ai/pipeline.test.ts`
  - `sanitizeForbiddenTokens` 6 단위 케이스 (require/globalThis/import/Function/process/clean).
  - `generateSceneCode` 가 gpt-4o 실패 샘플(`const fs = require('fs');` + `globalThis.useCurrentFrame()`)
    을 받아도 sanitize 후 통과함을 모의 sandbox 로 검증.
  - 잔존 forbidden 토큰(`eval`)이면 `SceneSandboxError` 가 type-narrow 가능하게 throw 됨을 검증.
  - 멀티-스텝이 outline 단계에서 throw 했을 때 `generateAssetMultiStepAsApiResponse` 가 dynamic-import 로
    `generateAsset` 를 호출해 fallback 응답을 반환함을 jest `doMock` 으로 검증.
- 신규 6 테스트, 총 39 passed (기존 33 → 39, 회귀 0).

## 왜 / 배경

TM-108 retro([[wiki/05-reports/2026-04-27-TM-108-quality-benchmark]]) 에 따르면 full(`AI_MULTI_STEP=1`)
경로는 gpt-4o 모델에서 4/5 케이스 sandbox 위반 → 5/5 사용자 측 500. Anthropic Sonnet 에서는 5/5 정상 — 즉
실패는 코드 결함이 아닌 **모델 출력 위생 차이** 에 의한 가용성 붕괴이며, 정책상 sandbox 를 풀어줄 수 없으므로
**LLM 이후 단계에서 입력을 정규화** 해야 한다는 결론이 나왔다.

또한 sanitize 가 모든 패턴을 막을 수 있다는 보장이 없으므로(예: encoded payload, 새 변종) 단일-샷 fallback 을
보험으로 둔다. 단일-샷 경로는 TM-100 의 fallback asset 까지 포함해 이미 가용성이 검증된 path 다.

## 영향

### 코드 / 시스템

- `pipeline.ts` 코드량 ≈ +110 lines, public API 추가(`sanitizeForbiddenTokens`, `SceneSandboxError`).
- `composedCode` 의 정의가 "scrubbed 버전" 으로 미세 변경됨 — 기존 호출자 중
  `MultiStepResult.composedCode` 에서 raw 형식을 가정하는 곳은 없음(테스트로 검증).
- `generateAssetMultiStepAsApiResponse` 의 응답 shape 에 `multiStep.fallback?: 'single-shot'` 추가
  (선택 필드, 기존 호출자 비파괴).

### 사용자 / 제품

- gpt-4o 환경에서 multi-step 활성화 시 가용성이 20% → ≥ 80% 수준으로 회복(추정, 단위 테스트 기반).
  최악의 경우라도 단일-샷 fallback 으로 절대 500 을 띄우지 않음.
- fallback 발동 시 사용자에게 `warning` 토스트로 가시화 — silent regression 방지.

### 비용 / 성능

- sanitize 는 정규식 6 회, 코드량 1-3 KB 기준 1ms 미만. 실측 latency 영향 없음.
- 단일-샷 fallback 발동 시 비용은 multi-step 실패 비용 + single-shot 비용 — 평균 ~1.7×.
  하지만 실패가 줄어들수록 호출 빈도도 감소하므로 dollar-weighted 영향은 크지 않을 것으로 예상.
