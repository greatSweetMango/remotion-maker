# TM-76 — Retrospective

## TL;DR

ADR-0010 fork lineage **passes live verification clean** (11/11 steps, 0 bugs). Discovered + fixed an infra footgun: worktree dev servers were silently sharing the main-repo SQLite DB because `DATABASE_URL=file:./dev.db` resolves against turbopack's inferred workspace root (the *parent* lockfile), not the worktree. setup-worktree.sh now pins absolute path.

## 잘 된 것

- Single-script E2E (`scripts/qa/tm-76-asset-lifecycle.mjs`) covers all flow stages from generate→edit→share→fork→authz→soft-delete→hard-delete cascade→orphan SQL check, with structured `summary.json` output. Reusable pattern for future schema lifecycle audits.
- Cross-user auth (FREE attempting to share/delete a PRO-owned fork) returns clean 403 on both endpoints, with no row mutation post-attempt.
- Cascade rules (`onDelete: Cascade` on AssetVersion, `onDelete: SetNull` on `Asset.sourceAsset`) behave exactly as schema declares.

## 문제와 해결

### Worktree DB shadowing (HIGH severity, infra)

**Symptom**: Newly-seeded users via `Prisma` in the QA script were invisible to the dev server's `authorize()` callback — login redirected to `?error=CredentialsSignin`.

**Root cause**: `.env.local` contained `DATABASE_URL=file:./dev.db`. Next.js 16 + turbopack detected two `package-lock.json` files (worktree root + main repo) and inferred the **main repo** as the workspace root. The relative SQLite path was thus resolved against the main repo's CWD, so the dev server opened `/Users/.../remotion-maker/prisma/dev.db` instead of the worktree's. The QA script (using its own `PrismaClient`) wrote to the worktree DB; dev server read main repo DB; they never met.

**Fix**: `scripts/setup-worktree.sh` now writes `DATABASE_URL=file:<absolute worktree path>/prisma/dev.db` into `.env.local` and uses the absolute URL for `prisma db push` too.

**Tech-note**: `wiki/02-dev/tech-notes/2026-04-27-worktree-database-url.md`

### Login flow research

NextAuth credentials sign-in via curl is non-trivial (CSRF cookie + form-encoded body + manual redirect). Captured the working pattern in the QA script's `login()` helper for future reuse.
