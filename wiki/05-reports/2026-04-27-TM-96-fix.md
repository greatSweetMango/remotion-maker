# TM-96 — Prisma `main.User does not exist` on main dev server

## Status

- **Severity**: P0 (user-blocking — `/dev-login` and all auth flows broken on main)
- **Verdict**: APPROVE — fix shipped, root cause confirmed, defense-in-depth in place
- **Confidence**: 95%
- **Branch**: `TM-96-prisma-db-mismatch`

## Reproduction

1. Main repo dev server running on the user's machine.
2. Visit `/dev-login` → `devAutoLogin` calls `prisma.user.findUnique` →
   `PrismaClientKnownRequestError: The table 'main.User' does not exist`.

Yet `sqlite3 prisma/dev.db ".tables"` lists all 8 tables with rows.

## Diagnosis (5 min)

```
lsof | grep dev.db
→ node 78437 fd 49u  REG  …/node_modules/.prisma/client/dev.db   (0 bytes)
```

The dev server's Prisma engine had opened **the wrong file** — an empty
SQLite DB at `node_modules/.prisma/client/dev.db` instead of the real one at
`prisma/dev.db`. `.env` / `.env.local` both had `DATABASE_URL="file:./dev.db"`
(relative), and the engine's CWD drifted to the generated-client directory.

Confirmed two `dev.db` files on disk:

| Path | Size | Tables |
|---|---|---|
| `prisma/dev.db` | 2,355,200 B | 8 |
| `node_modules/.prisma/client/dev.db` | 0 B | 0 |

This matches the Next 16 + Turbopack workspace-root inference class of bug
(TM-76 retro), and is adjacent to TM-87's worktree prisma-client cross-poll
fix. Main was the last place still using relative `file:./dev.db`.

## Fix

### Code (PR)

`src/lib/db/prisma.ts` — runtime DATABASE_URL normalization:

- If `DATABASE_URL` begins with `file:` and the path is relative, resolve it
  against `process.cwd()` (project root, in Next server context) and rewrite
  to absolute **before** instantiating `PrismaClient`.
- Log warning when a rogue empty DB is detected at the wrong CWD.
- No-op for absolute paths and non-sqlite providers.

This makes the bug self-healing even if someone writes `file:./dev.db` again.

### Local main repo (immediate user unblock, no commit)

`.env` and `.env.local` are gitignored, so the immediate fix is local:

```
DATABASE_URL="file:/Users/kimjaehyuk/Desktop/remotion-maker/prisma/dev.db"
```

Backups saved at `.env.bak` / `.env.local.bak`. Rogue empty file deleted:

```
rm /Users/kimjaehyuk/Desktop/remotion-maker/node_modules/.prisma/client/dev.db
```

User must **restart their dev server** for the new env vars to load.

## Verification

- `npx prisma db pull --schema prisma/schema.prisma --print` against the
  pinned absolute URL → emits a schema with all 8 models. ✓
- `sqlite3 prisma/dev.db "SELECT COUNT(*) FROM User;"` → `1` row exists. ✓
- After restart, `/dev-login` flow should call `findUnique({email})`,
  hit the existing dev user, and `signIn` succeeds.

## Retrospective

### What went right

- `lsof` diagnosis nailed the root cause in <2 minutes — fd-based truth-ground
  beats config-archaeology every time.
- Pre-existing TM-76 / TM-87 retros + the `setup-worktree.sh` comments
  pre-cached the failure mode mentally; recognition was instant.
- Two-layer fix (code guard + local env pin) means the bug can't easily come
  back even if a future contributor writes a relative URL.

### What went wrong

- The TM-76 fix only covered worktrees, not main. The `setup-worktree.sh`
  comment at line 102–106 documents the danger but the main-repo `.env` was
  left relative for ~9 days. We should have generalized the pin earlier.
- TM-87 `cp -R` of `node_modules/.prisma/client` from main to worktrees
  arguably increased the chance of stale or path-confused generated clients
  drifting between repos. The runtime guard now masks this.

### Action items

- [x] Add runtime DATABASE_URL normalization in `src/lib/db/prisma.ts` (TM-96)
- [x] Pin local main `.env` / `.env.local` to absolute path
