# TM-83 retro — perf concurrent driver

## What we did

- Built `__tests__/perf/tm-83-concurrent.mjs`: Node-22-fetch driver, no external deps.
- Measured 10 concurrent + 30 sequential POSTs to `/api/generate` against a freshly-booted dev server (PORT=3083).
- Captured RSS deltas, p50/p90/p99, status-code histogram, race-check signals.
- Wrote findings + numbers to `wiki/05-reports/2026-04-27-TM-83-perf.md`.

## What worked

- Single Node script, no axios/k6: ~250 lines, easy to inspect, no install. Sticking to `fetch` + `performance.now()` was the right call.
- Two-batch design (concurrent then sequential, same session token) made the TOCTOU quota race **fall out for free** on the first FREE-tier run. Wouldn't have caught it with N=1 sequential.
- Recording RSS via `lsof + ps` (no agent in the dev process) — simple, repeatable, no instrumentation in app code.

## What hurt

- **Worktree DB resolution.** First two boot attempts went to `node_modules/.prisma/client/dev.db` (a stray empty SQLite file Prisma created relative to the generated client install path, not the schema directory). Fix: pin `DATABASE_URL` to an absolute path in `.env.local`. **TM-87 supposedly isolated this; it did not.** This deserves a tech-note follow-up.
- **Stateless Bash cwd.** Several restart attempts launched `next dev` from the main repo root, not the worktree, despite the prefix. We ended up using `bash -c 'cd ... && npm run dev'`. Worth folding into `scripts/setup-worktree.sh`.
- **FREE tier limit=3 on dev user.** First batch was wasted (caught the race, but couldn't measure perf). Re-ran on PRO. Mild cost overrun (~$2.5 instead of $2).
- No node_modules in the worktree (next dev silently used the main repo's install). Worked, but is fragile — version drift between worktrees would break this. Out of scope to fix here.

## Things to remember

- **`/api/generate` quota check is TOCTOU-unsafe** — see perf report finding #1. The remediation (atomic conditional update) is small and we recommend a follow-up task.
- gpt-4o-mini wall ≈ 5–7s per call; concurrent fan-out gives ~28× wall-clock improvement but only ~11% per-request median improvement. Server is not the bottleneck.
- Dev-server RSS grew +119 MB / 40 calls and did not retract in 1.5s cooldown. Likely warming caches, not a leak; re-measure on `next start` build before alarming.
