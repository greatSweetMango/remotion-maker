# TM-95 — dev auto-login P0 fix

## Symptom

User-reported regression: in worktree dev environments with `DEV_AUTO_LOGIN=true`,
the dev auto-login flow no longer reliably logged the user in. Symptoms varied:

- Sometimes the user landed on `/dev-login` and stayed there ("Logging in..." text,
  blank afterward) without ever reaching `/studio`.
- Sometimes the user reached `/studio` but every userId-scoped query came back
  empty (no assets, can't save, etc.) — a confusing "logged-in-but-not-really"
  state.

## Root cause(s)

Two compounding bugs:

### 1. Client-JS-dependent auto-login is fragile

`src/app/dev-login/page.tsx` rendered an `<AutoLoginForm>` (client component)
which used `useEffect → ref.current?.requestSubmit()` to fire a Server Action
(`devAutoLogin`). This requires:

- React hydration to complete on the page
- The `useEffect` to actually run (React 19 strict-mode in dev double-invokes
  effects; if hydration fails or another error short-circuits, the submit may
  be canceled silently)
- The browser to have JS enabled and not have `prefers-reduced-data` style
  blockers

Empirical observation in this fix's reproduction: a curl-only request followed
the redirect to `/dev-login` and got a 200 page back, but **no POST to the
server action was ever issued** — the dev server log showed only GETs. With a
real browser the form *usually* submitted, but the path is non-deterministic.

### 2. Ghost session after worktree DB reset

NextAuth is configured with `session: { strategy: 'jwt' }`. JWTs are signed
with `NEXTAUTH_SECRET` and validated cryptographically — they survive even when
the underlying user row is deleted. Worktree bootstrap (`scripts/setup-worktree.sh`)
runs `prisma db push` on a fresh `dev.db`, which wipes the User table. Any JWT
cookie left over in the browser from a previous session then points to a
non-existent `user_id`.

The session callback in `src/lib/auth.ts` did:

```ts
session.user.id = token.sub;
const dbUser = await prisma.user.findUnique({ where: { id: token.sub } });
if (dbUser) session.user.tier = dbUser.tier;
return session;
```

If `dbUser` was null (deleted), the callback still returned a session with
`user.id` set — middleware saw a truthy session and let the request through to
`/studio`, where every `prisma.asset.findFirst({ where: { userId } })` returned
null. The user appeared logged in but every feature was broken.

## Fix

Three coordinated changes:

### `src/proxy.ts`

- When `DEV_AUTO_LOGIN=true` and the user is unauthenticated, redirect to the
  **server-side API route** `/api/dev/auto-login?callbackUrl=...` instead of
  the client-rendered `/dev-login` page. The API route performs `signIn()`
  server-side — no JS required — and the redirect chain is `307 → 307 → 200`
  in a single curl call.
- Treat `!session?.user` (not just `!session`) as unauthenticated, so a ghost
  JWT (session present but user invalidated by the auth callback) triggers
  re-login.

### `src/lib/auth.ts`

- In the `session` callback, if the JWT's `sub` does not resolve to a current
  DB user, return the session with `user: undefined`. Combined with the proxy
  change above, this self-heals after a worktree DB reset: stale JWT →
  middleware detects no user → redirects to `/api/dev/auto-login` → upserts
  the dev user row → fresh JWT issued → /studio renders.

### `src/app/api/dev/auto-login/route.ts`

- Add `tier: 'PRO'` to the dev user upsert (was missing — the parallel
  `actions.ts` already had it, but the API route didn't, causing the dev user
  to come up as `FREE` and hit feature gates).
- Update existing user's tier to `PRO` if it drifted.
- Add unreachable `NextResponse.redirect` after `signIn` to satisfy TS return
  type (signIn throws `NEXT_REDIRECT`).

## Verification

End-to-end live curl flow against `http://127.0.0.1:3095`:

```text
STEP 1 (no cookies): GET /studio
  → 307 to /api/dev/auto-login?callbackUrl=%2Fstudio
  → 307 to /studio (sets authjs.csrf-token + authjs.session-token)
  → 200
final:http://127.0.0.1:3095/studio code:200 numredir:2
post-state: User.email=dev@localhost, tier=PRO

STEP 2 (ghost JWT, empty User table): cookie=stale-token, DELETE FROM User
  → 307 to /api/dev/auto-login (proxy detects !session.user)
  → 307 to /studio (re-creates dev user, issues fresh JWT)
  → 200
final:http://127.0.0.1:3095/studio code:200 numredir:2
post-state: User row restored, tier=PRO
```

Both flows complete without any client-side JavaScript. No `useEffect`, no
hydration race, no chance of a stuck "Logging in..." page.

## Impact

- Removes a P0 regression blocking dev workflow in worktrees.
- Self-healing: dropping `prisma/dev.db` no longer requires manual cookie
  clearing in the browser.
- API route `/api/dev/auto-login` is now the canonical entry point;
  `/dev-login` page is left in place for backwards-compat but no longer on the
  hot path.
