# TM-112 — Retro

## 잘 된 점

- **RCA 가 빨랐다.** TM-108 r2 보고서에 "composer 측 PARAMS 출력 형식 표준화 또는 evaluator 측 식별자 추출 보강" 이라고 두 갈래가 명시돼 있었고, 코드를 5 분 안에 읽어 composer 측 가정(`Scene{N}Params` 가 LLM 출력에 *반드시* 존재) 이 근본 원인임을 확정.
- **단위 테스트로 RCA 를 박제**한 뒤 fix. 4 가지 LLM drift 패턴(중복 이름, 누락, 비-canonical, IIFE) 을 케이스화해 회귀 방지선 확보.
- **라이브 검증을 환경 이슈에 막히지 않게 우회**. /api/generate 가 prisma schema drift 로 500 을 내자, 파이프라인을 직접 호출하는 스크립트로 RCA 가 살아 있음을 확인. 평가자가 fix 의 유효성을 보고서 한 줄 ("pass=2/2") 로 검증할 수 있음.

## 어려웠던 점

- **dev 서버 환경 셋업이 초기에 막힘** — worktree 에 `.env.local` symlink 누락, `NEXTAUTH_URL` 이 3000 하드코드여서 PORT=3112 redirect 가 깨짐. Both 해결 후에도 prisma schema drift 발견. → 향후 `/orchestrate` worktree 셋업 단계에서 .env.local symlink + NEXTAUTH_URL override 를 자동화하면 다음 라이브 검증이 30 분 절약됨.
- **LLM 출력 패턴을 직접 못 본 채 RCA**. 벤치 결과 산출물(jsCode) 이 git 에 없었음 → 추정만으로 fix 설계. 다행히 라이브 검증에서 통과했지만, 다음엔 TM-108 r2 의 실제 jsCode 한 케이스만이라도 wiki 에 첨부하는 걸 권고.

## 배움 / 약속

- **LLM 의 naming convention 에 의존하지 말라** — system prompt 에서 강제해도 어긋남. wrapper/composer 는 항상 fragment 를 introspect 해 실제 이름을 사용해야 한다 (이번 TM-112 이전에도 evaluator-params-bug 로 같은 교훈을 얻은 바 있음 → 패턴 인식).
- **schema drift 와 fix-RCA 를 분리해 평가**. dev DB 가 다른 worktree 와 공유되면서 schema 가 어긋날 수 있다. 향후 worktree 셋업 시 prisma `db push` 를 자동화하거나, fix 검증 스크립트는 가능하면 DB 를 거치지 않게 설계.

## 메트릭

- 작업 시간: ~ 90 분 (worktree 셋업 30 분, RCA + fix 30 분, 단위/라이브 검증 30 분).
- 비용: gpt-4o full multi-step 2 회 ≈ $0.4 (judge 미사용).
- 회귀 안전망: pipeline.test.ts 4 종 신규 + 기존 39 → 43 통과.
