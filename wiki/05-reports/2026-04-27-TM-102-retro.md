# TM-102 retro

## What went well

- ADR-driven sequencing: writing the ADR placeholder first crystallized
  the cost-warning policy + rollout phases before any code.
- Reusing the gated-live-test pattern from TM-74 sidestepped the missing
  `tsx` runner — no new dev dep, runs through existing Jest config.
- Live smoke succeeded on 5/5 prompts both modes, no parser/transpile
  fallout from the per-scene compose path.
- Feature flag (`AI_MULTI_STEP`) keeps the production path untouched.
  Existing `generate.test.ts` (19/19) stayed green without modification.

## What hurt

- 2.3× wall time on multi-step is a real UX regression even with the
  cost warning — needs latency mitigation (cache reuse, parallel scene
  pipelines already in place but Anthropic side serialized somewhat).
- The `composeSceneCodes` function generates a `<Sequence>` timeline but
  inherits each scene's `<AbsoluteFill>` background — visual layering
  for scene transitions hasn't been validated end-to-end (only at the
  module-validation level).
- Code length (+16%) is a weak proxy for "Artifact-grade" — the bench
  (TM-46 r7) is where the real verdict lands. Don't celebrate yet.

## What I'd do differently

- Build the composer-level visual smoke test (snapshot the first frame
  of each scene via the Remotion player) before the LLM live run, so
  layering bugs surface before token spend.
- Surface per-stage token usage in `spend.json` so the cost-ratio
  warning can use measured data instead of the empirical formula.

## Triggers re-qualify

- TM-46 (visual-judge bench gate) — pipeline alters scoring distribution.
