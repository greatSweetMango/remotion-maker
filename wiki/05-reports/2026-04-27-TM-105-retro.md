# TM-105 회고

## 잘 된 것

- **단일 책임 분리**가 직관적이었다. mode 결정(=permissive, generate-favoring) 과
  question 생성(=expansive, axis-explorative) 은 본래 다른 task 인데 한 시스템
  프롬프트로 묶여 있었던 게 진짜 문제였다. 분리하니 둘 다 명확해졌다.
- 기존 UI(`ClarifyCard`) 가 이미 N×M 동적 렌더링을 지원하고 있어 백엔드만
  바꾸면 됐다. UI 회귀 zero.
- clarify-gate(TM-52/TM-68) 의 over-trigger 가드와 충돌 없음 — 그 경로는 forced
  generate 가 우선이고 dynamic question gen 은 그 가드를 통과한 vague 케이스에서만
  동작.
- live smoke 결과가 즉시 만족스러웠다 (logo / 추상적 / 슬라이드쇼 모두 서로 다른
  tailored 질문 → generic data_kind 사라짐).

## 아쉬운 것

- 원본 generate.ts 에 두 군데의 `return first.value` 가 있었고 (forced retry 후의
  return 포함), 첫 한 곳만 수정하는 결정이었다. 명시적 코멘트로 의도를 박아뒀지만,
  나중에 누군가 forced retry path 도 dynamic gen 을 원할 때 분기점이 한 군데 더
  생긴다.
- 예시 프롬프트 "이미지 슬라이드쇼" 는 clarify-gate 의 STYLE_PATTERNS("슬라이드")
  에 걸려 over-trigger 로 분류된다. 이는 별개 이슈지만, TM-105 demo 효과는 약간
  희석. (TM-52 정책 합당함, 데모만 약간 헷갈림.)

## 배운 것

- few-shot prompt 의 example 다양성이 결정적. 단 하나의 예시는 LLM 을 그 예시에
  완전히 anchor 시킨다. 3개 이상의 distinct 예시가 axis 다양성을 가르친다.
- multi-step LLM pipeline 은 비용/지연 대비 품질 ROI 가 좋은 도구. clarify 처럼
  사용자가 어차피 멈춰 있는 구간이면 latency 비용이 거의 sunk cost.
- normalize/validate layer 는 LLM I/O 라이브러리에서 필수. dedupe + drop-malformed
  + min-count 보장 셋이 안전 그물.

## 다음에 할 것

  refactor 수준). 향후 다른 LLM-heavy 작업에서 이 패턴 재등장하면 ADR 승격.
- clarify rate 측정 — `.agent-state/spend.json` 콜 빈도와 함께 생산 환경 clarify
