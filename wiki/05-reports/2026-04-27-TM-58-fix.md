# TM-58 — 프롬프트 길이 cap

## 한 줄 요약

`/api/generate` + `/api/edit` 가 10,500자 prompt 를 200 OK 로 받던 갭을 닫았다. **2,000자 cap**, 400 + `PROMPT_TOO_LONG` 코드, 사용자향 한국어 메시지. DB / 쿼터 / LLM 호출 **이전** 차단 → 비용 증폭 0.

## 배경

[[05-reports/2026-04-27-TM-45-edge-fuzzing|TM-45 fuzz]] 의 B3 케이스 (10,500 chars) 가 200 OK 로 통과 → 비용 amplification + context window 포화 abuse vector. 입력 검증 갭 3건 중 medium priority 항목.

## 변경 사항

### 신설

- `src/lib/validation/prompt.ts` — `MAX_PROMPT_LENGTH = 2000`, `validatePrompt()` 공유 유틸
  - 빈/공백/비-string → `PROMPT_REQUIRED`
  - 길이 초과 → `PROMPT_TOO_LONG` + `meta: { length, max }`
  - 한국어 메시지: `프롬프트가 너무 깁니다. 2000자 이하로 입력해주세요. (현재 N자)`

### 수정

- `src/app/api/generate/route.ts` — `validatePrompt()` 호출을 **DB lookup 보다 먼저** 배치 (쿼터 burn 방지)
- `src/app/api/edit/route.ts` — 기존 `assetId/prompt/currentCode` 필수 체크 직후 `validatePrompt()` 추가

### 테스트 (13 added, 0 regression)

- `__tests__/lib/validation-prompt.test.ts` (6)
- `__tests__/api/generate/length-cap.test.ts` (5) — 10,500자 → 400, `generateAsset` not called, DB not touched, cap 경계값
- `__tests__/api/edit/length-cap.test.ts` (2)

## Cap 값 결정 (2,000자)

- 한국어 멀티장면 상세 프롬프트 평균 200-600자 → 4-10배 헤드룸
- TM-45 B1 (2,160자) 도 합법 케이스로 보였으므로 살짝 좁힘 — 2,000 미만이면 전부 정상 통과
- 추후 PRO 차등이 필요하면 ADR 동반하여 4,000/8,000 분기 가능. 현재는 단일 cap 으로 단순화.

## 보안 효과

| 시나리오 | 이전 | 이후 |
|---|---|---|
| 10,500자 prompt | 200 OK, LLM 호출됨, 쿼터 차감 | 400 `PROMPT_TOO_LONG`, LLM 호출 X |
| 100,000자 flood | 1 호출당 토큰 폭증 | 동일 (전부 400) |
| 정상 한국어 멀티장면 (~600자) | OK | OK (변화 없음) |
