# TM-93 회고

## 잘 된 점

1. **TM-92 retro 의 follow-up §2.3 → 1 hop 으로 fix.** "editUsage 도
   audit 필요" 가 retro 에 박혀 있었기 때문에 별도 발견 단계 없이 바로
   route 코드 검토 → race 확인 → 픽스 가능. tech-note + retro 가 다음
   회차의 input 으로 직결된 두 번째 사례 (TM-83→TM-92 에 이어).
2. **DB migration 회피.** `editUsage` 가 JSON column 이라는 제약 안에서
   SQLite JSON1 (`json_extract` / `json_set`) + single-statement
   atomic UPDATE 로 race 종결. schema 변경 없음 → escalate 트리거 안
   걸림 → 자율 실행 유지.
3. **회귀 테스트가 race 자체 + injection 표면 둘 다 검증.** TM-92 의
   mock-mutex 패턴을 그대로 차용 (5 → 3 인터폴레이션 차이만). 추가로
   `assetId` 화이트리스트 위반 케이스로 raw SQL/JSON-path injection
   surface 도 명시적 테스트.

## 아쉬운 점

1. **mock 기반 race 테스트의 한계 (TM-92 와 동일).** 진짜 SQLite +
   Prisma 의 race 는 integration 테스트가 있어야 100% 안전. TM-92 후속
   "tm-83-concurrent.mjs 를 jest+SQLite 로" 가 아직 미진행 — 이 task
   에서도 같은 limitation 그대로 안고 있음.
2. **`assetId` 식별자 화이트리스트 = SQL injection 차단의 *유일* 라인.**
   Prisma tagged template 은 식별자 자리(JSON path key)를 parameter
   binding 으로 받지 못해서 직접 interpolate 했다. cuid 는 이미
   `[a-z0-9]+` 라 안전하지만, 템플릿 prefix 가 바뀌거나 새 ID 형식이
   추가될 경우 화이트리스트도 같이 갱신해야 한다 (코드 주석에 명시).
3. **헬퍼 분산.** `reserveEditSlot`/`refundEditSlot` 만 `usage.ts` 로
   추출했고, TM-92 의 `monthlyUsage` reserve/refund 는 여전히
   `route.ts` 에 inline. 두 quota 게이트 헬퍼를 같은 파일에 정렬하는
   refactor 가 다음 후속 가치 있음 (별도 task 권장).

## 다음에 시도

1. **integration race 테스트 (TM-92 retro 와 같은 권고 재확인).**
   `__tests__/perf/tm-83-concurrent.mjs` 를 jest 의 격리 SQLite DB +
   prisma push 와 묶어 진짜 race 회귀를 매 PR 잡기. /api/generate +
   /api/edit 두 entry 모두 커버.
2. **`reserveQuotaSlot` 통합 헬퍼.** `monthlyUsage` (Int) 와
   `editUsage` (JSON) 두 모드를 한 모듈에 두고 route 는 호출만. 환불
   누락 / 분기 누락 위험 감소.
3. **upload 카운터 audit.** TM-92 retro §3 "다른 quota 게이트 점검"
   의 마지막 항목. uploaded asset 카운터도 read-modify-write 라면
   같은 fix 패턴 적용 검토.
