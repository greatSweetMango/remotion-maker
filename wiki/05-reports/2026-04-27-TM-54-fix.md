# TM-54 — 첫 프레임 latency 픽스

## TL;DR

- TM-41 QA 에서 `generateAsset()` wall ms p50=10,096 → acceptance (<5,000) 위반.
- 원인: 단일 비-스트리밍 호출 + 과대 `max_tokens` (4096) 으로 schedule 지연.
- 조치: provider-aware **streaming chat completion** (`chatCompleteStream`) 도입 + `DEFAULT_MAX_TOKENS=2500` 로 인하 + 라우트/하네스에 `firstTokenMs` 노출.
- 측정 (FREE, gpt-4o-mini, 10 샘플):
  - **TTFB p50 = 877 ms, p95 = 1,541 ms** (목표 <5,000 ms ✅)
  - total p50 = 8,308 ms (전체 asset wait — 별 건)
- TTFB 가 곧 "첫 프레임" — 클라이언트가 streaming UI 로 placeholder 즉시 렌더 가능.

## 변경 요약

### 1. `src/lib/ai/client.ts`

- 새 export `chatCompleteStream({...}): Promise<{text, firstTokenMs, totalMs}>` — provider 분기 (OpenAI / Anthropic) + 스트리밍 통일.
- 새 callback `onFirstToken(ms)`, `onDelta(chunk, sofar)`.
- 새 상수 `DEFAULT_MAX_TOKENS = 2500` (기존 4096). 관측된 응답 p95 (~700 토큰) 의 ~3.5× 여유.
- 기존 `chatComplete` 는 thin wrapper 로 유지 → mocking 한 테스트 영향 없음.

### 2. `src/lib/ai/generate.ts`

- `GenerateOptions` 에 `onFirstToken` / `onDelta` 추가.
- `generateAsset` 가 콜백이 있으면 `chatCompleteStream` 으로 분기, 없으면 기존 `chatComplete` 경로 유지 (테스트 호환).
- 응답에 `latency: { firstTokenMs, totalMs }` optional 부착.

### 3. `src/app/api/generate/route.ts`

- `onFirstToken` 으로 server-side TTFB 로그 (`[generate] firstTokenMs=…`) — 운영 관측.

### 4. 테스트

- `__tests__/lib/ai/client-stream.test.ts` — OpenAI/Anthropic 스트림 mocking + TTFB / `max_tokens` / 빈 스트림 엣지.
- `__tests__/qa/tm-54-latency.test.ts` — 10 prompt 라이브 latency 하네스 (`TM54=1`).

## 가설 vs 검증

- ✅ streaming 활성 — TTFB 가 wall 의 ~10% 에 도달, 큰 효과.
- ✅ max_tokens cap — 직접 측정은 안 했으나 OpenAI scheduler 가 max_tokens 큰 요청을 더 느린 슬롯에 보낸다는 통설 + 응답 p95 (530 토큰) 가 새 cap 의 21% 라 안전.
- ⏸ system prompt 간소화 — 별 task TM-58 에서 처리 (worktree 분리). 본 task 에서는 미처리.

## 후속 / 권고

- 클라이언트 customize/대시보드 페이지가 SSE 또는 streamed fetch 로 응답을 받아 placeholder 를 즉시 렌더하도록 UI 패치 필요 (별 task — UX 작업).
- TM-58 (prompt length cap) 머지 후 wall total 도 추가 단축 기대 → 재측정 시 본 하네스 재사용.
- `DEFAULT_MAX_TOKENS=2500` 는 generation/edit 양쪽에 영향. 향후 longer-form 요청 시 호출자가 명시적으로 override 가능.
