## 시나리오

### 1) 비로그인 → 401 (5/5 PASS)

쿠키/세션 없이 `POST /api/generate`. 5회 모두 `HTTP 401 {"error":"Unauthorized"}` 반환. middleware (proxy.ts) 는 `/api/*` 매처에서 제외되어 있으나 라우트 핸들러 자체에서 `auth()` 검사 후 401 응답. 우회 경로 없음.

### 2) FREE — 쿼터 허용 + 차단 (5/5 PASS)

- 사전 SQL: `UPDATE User SET tier='FREE', monthlyUsage=0`
- 쿼터 미만 호출 (2회): 첫 호출 `type=clarify` (정책상 quota 차감 X), 두 번째 호출 `type=generate` (usage 0→1). 모델 로그: `tier=FREE model=gpt-4o-mini`
- 쿼터 소진 SQL: `UPDATE User SET monthlyUsage=3` (FREE limit=3)
- 차단 호출 (3회): 모두 `HTTP 429 {"error":"Monthly generation limit reached (3). Upgrade to Pro for 200/month."}` 반환. 메시지 명확, 업그레이드 경로 안내 포함.

### 3) PRO — 4o 라우팅 (5/5 PASS)

- 사전 SQL: `UPDATE User SET tier='PRO', monthlyUsage=0`
- 5개 프롬프트 (단순/복잡 혼합) 모두 `HTTP 200 type=generate`
- 모델 로그: `tier=PRO model=gpt-4o` (instrumentation 라인으로 확인)
- 5번째 호출 후 `monthlyUsage=5` → PRO limit=200 미달, 차단 미발생

### 4) Dev 자동로그인 우회 (PASS)

`DEV_AUTO_LOGIN=true` 인 상태에서도:
- `proxy.ts` 는 `/studio`, `/dashboard` 에만 적용 (api 경로는 matcher 제외)
- `/api/dev/auto-login` 라우트는 `NODE_ENV` 무관하게 env flag 만 검사 → flag false 시 404 (현재 환경은 true 이므로 dev 전용)
- API 게이팅은 라우트 핸들러 `auth()` 가 단독 책임. proxy.ts 가 미적용되는 점은 **의도된 분리** — API 는 매 요청마다 세션 확인.
- 비로그인으로 `/api/generate` 직접 호출 시 401 (시나리오 1에서 검증)

## 발견 / 코멘트

- **버그 없음**. 게이팅, 라우팅, 쿼터 메시지 모두 spec 부합.
- **clarify 응답이 quota 를 차감하지 않는 정책** (`route.ts` line 46-49) 은 의도적이며, 검증된 대로 동작 — 모호한 프롬프트로 무한 clarify 받아도 quota 소진 X.
- **proxy.ts 가 `/api/*` 미적용** — 라우트별 `auth()` 호출에 의존. 신규 API 라우트 추가 시 인증 누락 위험. 별도 ADR 또는 서버 미들웨어 helper 도입 검토 가치 있음 (현재 task 범위 외 — `triggers_requalify` 트리거 안 함).
- **주의**: Next.js Turbopack 이 root 를 부모 디렉터리(`/Users/kimjaehyuk/Desktop/remotion-maker`) 로 추론 → `DATABASE_URL=file:./dev.db` 가 부모 dev.db 를 가리킴. 격리 worktree dev 환경 운영 시 `turbopack.root` 명시가 안전. 본 task 의 `next.config.ts` 변경 범위는 아님.
