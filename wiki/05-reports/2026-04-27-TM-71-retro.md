# TM-71 회고 — visual-quality prompt-level pass

## 무엇을 했나

TM-46 r3 의 acceptance 미달(평균 71.2, 목표 75) 의 1차 원인이 prompt-level
guidance 부재라는 가설로, `GENERATION_SYSTEM_PROMPT` 에 카테고리별 명시
가이드라인 블록 (DATA-VIZ / TRANSITION / TEXT-ANIM / INFOGRAPHIC / LOADER) 을
추가. lock-in 단위 테스트 1건과 5-case offline 검증 스크립트를 함께 도입했다.

## 잘 된 것

- 변경 범위가 매우 작음 — `prompts.ts` 한 곳 (+~70 LoC). 호출 코드 무변경.
- 단위 테스트가 카테고리별 키워드(예: "Typography first", "midpoint frame",
  "x-axis", "RGB-split")를 바로 lock-in 해서, 향후 prompt refactor 가
  카테고리 가이드라인을 누락시키면 즉시 빨간불.
- 검증 스크립트(`tm-71-validate.ts`) 가 풀 capture+judge 파이프라인 없이도
  코드 길이 / PARAMS / JSX / motif 까지 빠르게 본다 — ~$0.05 / ~50s.
- 5-case 1차 run 5/5 성공. dv-03 의 4-slice pie chart, tr-02 의
  `interpolateColors(black→white)` 등 r3 의 직접 실패 모드가 prompt 만으로
  해소되는 것을 코드 레벨에서 확인.

## 아쉬운 것 / 학습

- 검증 스크립트 첫 작성 시 `generateAsset` 의 반환 타입(top-level vs
  `{kind, value}` 래핑) 을 잘못 추정해 5/5 모두 false-fail 로 잡혔다. 다음에
  `generateAsset` 처럼 union 반환 타입을 다룰 땐 호출 직후 `console.dir`
  한 번 떨궈 모양 확인할 것.
- 2차 run 에서 stochastic 으로 dv-02 (`require` security guard) / tr-08
  (transpile retry 실패) 가 나왔다. prompt 변경과 무관한 기존 모드지만,
  TM-71 검증 단계에서 처음 노출된 모드라 향후 별도 task 로 추적할 만함.
- 가이드라인이 길어진 만큼(+~2KB) cache hit 가 깨지면 비용 영향이 ~2x. 다만
  현재 ephemeral 캐시 적용 중이라 첫 호출 외엔 사실상 무영향.

## 다음에 더 잘 하려면

- TM-46 r4 풀 30-prompt 재실행으로 정량 효과 측정 후, 카테고리별 평균이
  여전히 부족하면 (a) few-shot 추가 또는 (b) 카테고리별 system prompt
  분기 ADR 검토.
- 검증 스크립트(tm-71-validate.ts) 를 generic 화해서 prompt change 시마다
  돌릴 수 있는 fast lane 으로 승격 검토 — 풀 r4 비용(~$0.6) 의 1/12.
- offline heuristic motif 만으론 fidelity (사용자가 의도한 색상/숫자 일치)
  를 측정 못함. judge 호출 1-2 케이스를 차후에 끼워넣을 수도.
