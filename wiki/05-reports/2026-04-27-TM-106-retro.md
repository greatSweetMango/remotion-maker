# TM-106 회고

## 결과

- 사용자 보고 P0: change/edit 정상 동작 X
- 의심 후보(TM-93/TM-100) 둘 다 라이브 검증 결과 회귀 아님 — 라우트 자체는 정상 동작
- 진짜 회귀 발견: 클라이언트가 `/api/edit` 응답의 `data.id` 를 무시 → template-backed asset의 모든 edit이 `assetId: "template-…"` 로 재전송 → editUsage path-key 누적, monthlyUsage 중복 차감, 새로고침 시 edit 손실
- fix: `ADD_VERSION` action에 `newAssetId`/`newTitle` 추가 + Studio.tsx URL 동기화
- 회귀 테스트 4개 신규 + 기존 슈트 모두 PASS

## 무엇이 잘 되었나

- Orchestrator 컨텍스트(suspect 후보)를 그대로 신뢰하지 않고 **라이브 재현부터** 시작 → 후보 두 개 모두 OK 확인. 잘못된 가설로 ripping 안 함.
- PRO + FREE 양쪽 토글로 quota cap도 같이 검증 → TM-93 atomic UPDATE 동작 재확인 (200×3, 429×2).
- root cause를 라우트 외부(클라이언트 reducer)에서 발견 — 라우트 라인만 보고 있었으면 못 잡았을 패턴.

## 무엇이 아쉬웠나

- 사용자 보고가 "정상 동작 X" 로 추상적이었음 → 라이브 재현이 길어짐. 다음엔 수동 STR 1줄이라도 받기.
- 현재 `/api/edit` 가 모든 template edit에서 `monthlyUsage +1` 차감 (route.ts L111). 1회 머터리얼라이즈에만 차감해야 의미적으로 맞음 — 별도 후속 권고로 남김.
- 새로 만든 e2e (Playwright)로 새로고침 시나리오 확인했으면 더 좋았을 것. 시간 budget로 단위 테스트 + 라이브 curl로 갈음.

## 시간/비용
