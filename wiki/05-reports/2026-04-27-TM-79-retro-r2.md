# TM-79 r2 Retrospective

## What we set out to do

Re-verify the 4 Player controls (pause / seek / loop / speed) across 3 templates (counter-animation, glowing-text, fluid-blobs) — total 12 cases — now that TM-99 (PR #93) has shipped the missing playback-rate UI flagged by r1.

## What went well

- **Rebase was clean** — TM-99 + 3 sibling PRs landed in main without merge fallout in this worktree.
- **r1 F1 finding (playback-rate missing) is fully resolved**. Live test confirms the TM-99 4-button toolbar reflects state via `aria-pressed` and feeds the Player's `playbackRate` prop. Spec contract is met.
- **Single-evaluate test pattern** for counter-animation got all 4 controls verified in one shot, avoiding repeated navigations that lose the page context. Worth keeping for future Player QA.
- **Robust seekbar selector**: discovered the Remotion seekbar can be located reliably by its inline-style signature (`cursor:pointer; touch-action:none; user-select:none`) — no dependency on classnames or DOM hierarchy, so it survives Remotion version bumps as long as the inline style is stable.

## What didn't go well

- **Cross-worktree browser session hijack (F2 from r1) recurred**. With TM-84-r2 also driving Playwright on :3084 concurrently, the shared MCP browser kept redirecting our tab off :3079 before glowing-text and fluid-blobs could be exercised live. Lost 5+ minutes to redirect-fight and one full dev-server crash.
- **8/12 cases not live-verified**. Substitute evidence (code-equivalence audit + sibling-worktree partial run) is reasonable but not as authoritative as a green run on the actual asset.
- **Dev server crashed mid-session** after a Remotion license-warning console-spam burst + a stale auth-session reload. Required a restart, costing ~90s.

## Root cause / structural notes

- Playwright MCP server is process-global within Claude Code; concurrent TeamLead agents share the same browser-context tabset. There is no isolation primitive at the agent boundary today.
- Sibling TM-84-r2 is mid-run on the same Playwright; its navigations are observable as our redirect chain ending in `localhost:3084/...`. This is not a bug in either worktree — it is an orchestrator-level scheduling assumption (concurrent QA agents are safe) that doesn't hold for shared MCP browsers.

## Action items (not blocking TM-79 close)

1. **`AI-INFRA-worktree-port-bleed`** (proposed r1, still not filed) — make Playwright session per-agent, or at minimum serialize QA runs that drive the MCP browser. Owner: orchestrator.
2. **PlayerPanel data-testids**: add a `data-testid="remotion-seekbar"` on a wrapper around the Remotion seekbar so future QA does not need the inline-style heuristic. Optional; current heuristic works.
3. **Document the single-evaluate pattern** in `wiki/02-dev/tech-notes/` so the next QA author can copy it instead of fighting per-step navigation flakiness.
