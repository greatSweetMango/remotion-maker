## 잘 된 점

1. **post-validate를 generate.ts 안에 두고 sandbox 와 분리** — sandbox 는 보안 deny-list, generate 는 quality 가드. 책임 분리 명확.
2. **휴리스틱 4개의 OR**: 단일 길이 체크가 아닌 PARAMS / JSX / `=> null` / length 다중 신호로 false-positive 줄임. 실제 250-char substantive code (`acceptance` 케이스) 가 통과하는 것 확인.
3. **retry 의 token cost 제어**: reinforcement 는 retry 일 때만 prepend → 정상 흐름 (95%+) 의 token 사용 변화 0. PRO stub 발생 시에만 약 +400 token.
4. **테스트가 실패 모드 그대로 재현**: canonical stub `() => null;` 을 그대로 mock 응답으로 넣어 retry 동작 검증. live QA 의존 없이 단위 테스트로 보장.

## 아쉬운 점

1. **live OpenAI 호출은 안 함**: 비용 ~$0.05 OK 라고 했지만, 단위 테스트가 충분히 cover 한다고 판단. TM-41 재실행 시 자연스럽게 검증.
2. **PARAMS 휴리스틱의 한계**: `const PARAMS = ...` regex 만 본다. 모델이 `const params = ...` (소문자) 로 응답하면 ADR-0002 자체 위반이지만 우리 가드는 통과 후 evaluator 단계에서 `extractParameters` 가 0개 추출 → 빈 customize. 이 케이스는 `extractParameters` 결과 0 도 placeholder 신호로 추가 검토 필요 (follow-up).
3. **dev console.warn**: log 가 sentry/telemetry 로 전달되지 않음. PRO 모델 placeholder 빈도가 prod 에서 어떻게 되는지 추적할 metric 미수립 (follow-up).

## 배운 것

- OpenAI gpt-4o 는 vague prompt + structured JSON schema 강제 시 **형식만 만족하는 minimal stub** 반환 패턴이 있음. Anthropic Claude haiku 는 같은 patterns 에서 substantive code 반환 (FREE 96% pass). prompt 변경보다 **post-validate + retry** 가 더 robust.
- `validateCode` (sandbox) 는 보안용이고 quality 보증 도구가 아님 — 별도의 quality gate 필요.

## Follow-up 후보 (별도 task 후보)
