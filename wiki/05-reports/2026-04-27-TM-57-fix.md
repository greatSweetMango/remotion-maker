# TM-57 — zero-width 프롬프트 우회 차단

## TL;DR

TM-45 fuzz harness 케이스 A4(zero-width 문자만으로 구성된 프롬프트)가 `prompt.trim().length === 0` 검사를 우회해 LLM에 빈 입력으로 흘러갔다. `validatePrompt`에서 `stripZeroWidth` 후 `.trim()` 으로 emptiness를 판정하도록 수정하고, regression 테스트 12건을 추가했다. 길이 cap 검사는 원본 `.length` 그대로 두어 invisible 패딩으로 cap을 우회할 수 없게 했다.

## 무엇이 바뀌었나

- `src/lib/validation/prompt.ts`
  - 새 `ZERO_WIDTH_RE = /[​-‍⁠﻿]/g` 상수 추가 (5종 invisible-format 문자 커버).
  - 새 `stripZeroWidth(input)` export — 외부 호출자가 정제된 프롬프트를 LLM에 넘길 때 재사용 가능.
  - `validatePrompt`의 emptiness 판정을 `stripZeroWidth(prompt).trim().length === 0` 으로 강화. PROMPT_REQUIRED 코드는 그대로 유지.
  - 길이 cap 검사는 원본 `prompt.length` 사용 → invisible 문자로 cap 우회 불가.
- `__tests__/lib/validation-prompt.test.ts`
  - zero-width handling describe 블록 추가 — ZWSP/ZWNJ/ZWJ/U+2060/U+FEFF 단독 입력, 혼합 whitespace, TM-45 case A4 재현, 길이 cap 우회 방어, `stripZeroWidth` 직접 검증 포함.
  - 전체 18 tests pass.

## 왜 / 배경

JavaScript `String.prototype.trim()` 은 Unicode 공백류만 제거하고 zero-width / format 카테고리 문자(U+200B–U+200D, U+2060, U+FEFF)는 건드리지 않는다. 사용자가 (또는 fuzzer 가) invisible 문자만 붙여넣으면 `.trim().length > 0` 이 되어 검증을 통과하고, 그대로 프롬프트로 전달되어 LLM 토큰 비용을 발생시킨다. TM-45 edge fuzzing 30 cases 의 case A4 가 정확히 이 패턴이었고 graceful 했지만 **차단되지 않은** 입력으로 보고됐다. (참조: `wiki/05-reports/2026-04-27-TM-45-edge-fuzzing.md` case A4)

## 영향

- 코드 / 시스템: `/api/generate`, `/api/edit` 양쪽이 공유하는 단일 validator 만 수정. 호출 경로 변경 없음.
- 사용자: 정상 사용에는 영향 없음 (visible 문자 + zero-width 혼합은 그대로 통과). zero-width-only 입력은 기존 PROMPT_REQUIRED 에러로 차단되며 이미 클라이언트가 처리 중인 코드 경로.
- 비용: invisible-only 프롬프트의 LLM round-trip(소액이지만 0이 아님) 0 으로 감소.
- 보안/abuse: 길이 cap 우회 시도(invisible 패딩) 도 동시에 봉쇄.
