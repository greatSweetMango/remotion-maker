# TM-95 retro

## Outcome

Fixed. Two compounding causes identified and addressed:
1. Client-JS-dependent auto-login flow (AutoLoginForm useEffect) replaced with
   server-side API route invocation from middleware.
2. Ghost JWT session (token survives worktree DB reset) detected and invalidated
   in the session callback; middleware now triggers re-login on `!session?.user`.

Verified end-to-end via curl-only flow — works without any browser JS.

## What went well

- **Reproducing the issue.** Both Playwright and curl reproductions converged
  on the same finding: no POST to the server action ever happened on the curl
  flow, and on Playwright the AutoLoginForm only sometimes fired. That made
  the JS-dependency bug obvious without needing a flaky-test report.
- **Ghost-session bug surfaced naturally** while debugging. A `User` table that
  stayed at 0 rows even after a "successful" `/studio` 200 was the smoking gun
  that pointed at the JWT-vs-DB mismatch. Easy to miss without DB inspection.
- **Server-side fix is more robust.** Removing the JS dependency removes a
  whole class of future bugs (hydration errors, strict-mode double-effects,
  concurrent renders).

## What could be better

- The original `AutoLoginForm` was an unnecessary indirection — the API GET
  route already existed but wasn't used. Should have been the first choice.
- Session callback that returns a user without verifying DB existence is a
  trap for any JWT-strategy NextAuth setup. Should be a default pattern.
- `prisma db push --force-reset` triggered a Prisma consent prompt; used a
  raw `DELETE FROM User` instead. Worth noting for future reset flows.
