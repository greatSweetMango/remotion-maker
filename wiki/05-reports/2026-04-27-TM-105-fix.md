# TM-105 — Dynamic clarify question generation

## 문제

GEN-06 (TM-11)부터 clarify 모드를 도입했지만, 사용자 보고로 "clarify 질문이
항상 같다"는 피드백이 누적되었다. 실제로 시스템 프롬프트의 few-shot 예시가
하나뿐이었고 (`데이터 종류` + sales/users/ranking), LLM 이 그 예시에 과도하게
앵커링되어 어떤 프롬프트든 비슷한 질문을 반환하는 경향을 보였다.

대표 사례:
- "이미지 슬라이드쇼" → `데이터 종류는?` (무관)
- "logo intro" → `데이터 종류는?` (무관)
- "뭐 좀 멋진거" → `데이터 종류는?` (무관)

## 결정

**multi-step 파이프라인 도입.** clarify 모드 결정은 기존 단일 호출에 두되,
"어떤 질문을 할 것인가"는 별도 LLM 호출로 분리한다.

이유:
1. 모드 결정과 질문 생성은 최적 프롬프트가 다르다. 모드 결정은 generate 우선,
   질문 생성은 expansive 한 axis 탐색.
2. 분리하면 독립적으로 튜닝/테스트 가능. 향후 question-gen 만 더 작은 모델로
   다운그레이드 가능.
3. 비용 영향이 작음. clarify 가 트리거된 케이스에서만 추가 1콜 (~$0.001-$0.003).

## 구현

### 새 모듈: `src/lib/ai/clarify-questions.ts`

- `generateClarifyQuestions(prompt, options)` — 전용 LLM 호출.
- `CLARIFY_QUESTION_GENERATOR_SYSTEM_PROMPT` — 풍부한 few-shot 예시 (slideshow /
  logo intro / 차트 3종) + axis 가이드 (subject, style, palette, pacing, text
  overlay, BGM mood, layout, data, transition, count).
- `normalizeClarifyResponse(raw)` — malformed 항목 drop, 3-5개 보장, dupe id 정리.
- `parseClarifyJson(text)` — fenced/prosed JSON 모두 처리.
- `ClarifyQuestionsValidationError` — 구조적 실패 시그널.

### 통합: `src/lib/ai/generate.ts`

primary 호출이 `mode=clarify` 를 반환하고 사용자가 answers 를 안 줬을 때:
1. `generateClarifyQuestions(prompt)` 호출.
2. 성공 시 `first.value.questions` 를 tailored set 으로 교체.
3. 실패 시 fallback (primary 의 질문 그대로). 사용자를 절대 차단하지 않는다.

TM-52/TM-68 over-trigger 가드는 그대로 유지. 그 경로는 "concrete prompt 인데
LLM 이 잘못 clarify 한 케이스" 로 forced retry 가 우선이고, 만약 여전히 clarify
면 generic 질문이라도 surface 하는 게 나음 (드문 edge).

### JSON 강제

OpenAI: `chatCompleteStream` 이 이미 `response_format: json_object` 강제 (TM-69).
Anthropic: 시스템 프롬프트의 strict 지시 + `parseClarifyJson` 의 fence stripping
+ balanced-brace recovery 로 처리.

### UI

`PromptPanel.tsx` 의 `ClarifyCard` 는 이미 N개의 questions/choices 를 동적으로
렌더하고 all-answered gating 을 제공. 따라서 UI 변경 없음. 한 번에 다 노출 +
선택형 칩 패턴 유지.

## 검증

### Unit (jest)

- `__tests__/lib/ai/clarify-questions.test.ts` — 16 cases (parse, normalize, generate
  happy/sad/edge).
- `__tests__/lib/ai/generate.test.ts` 갱신 — TM-105 wiring path 추가, 기존 clarify
  테스트는 fallback path 로 변경.
- `__tests__/lib/ai/clarify-gate.test.ts` 갱신 — vague clarify path 가 이제 2 콜
  (primary + dynamic).
- AI 디렉토리 전체: 164 passed / 5 skipped / 0 fail.

### Live (OpenAI gpt-4o-mini)

`scripts/tm-105-live-smoke.ts` 실행. 서로 다른 vague 프롬프트 3개에 대해
**완전히 다른** tailored 질문 세트 생산 확인:

```
"logo intro" → reveal_style / palette / duration / background_style
"뭐 좀 멋진거" → subject_kind / visual_style / pacing / bgm_mood
"이미지 슬라이드쇼" → over-trigger gate 가 forced-generate 로 처리 (clarify-gate 기존 정책)
```

generic `data_kind/sales/users/ranking` 더 이상 등장하지 않음.

## 영향 범위

## Cost / latency

- 추가 LLM 콜 1회 / clarify 응답 (clarify rate 는 TM-52/TM-68 가드 후 매우 낮음).
- gpt-4o-mini 기준 ~$0.001-$0.003.
- 추가 latency ~1-2 초 (사용자가 어차피 답하기 위해 멈추는 구간이라 체감 작음).
