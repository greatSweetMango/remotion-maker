# TM-45 회고 — 1회차

## 잘 된 것

- **Fuzz harness 를 plain Node ESM (`run.mjs`) 으로 작성** — `tsx`/`ts-node` 미설치 환경에서도 즉시 실행. 외부 deps 0개. 다른 QA task 가 같은 패턴 재사용 가능.
- **Sandbox `validateCode` 를 harness 안에 미러링**해서 200 OK 코드를 후검증. 이번에는 D8 가 이미 server-side 에서 차단되어 후검증이 발동할 일이 없었지만, 만약 LLM 이 deny-list 우회 표현(`Worker`+escape, `f‌etch` zero-width 분리)을 만들었다면 잡을 수 있었을 것.
- **NEXTAUTH_URL=3045 격리 패치** — main `.env.local` 이 3000 에 박혀 있어서 그대로 복사하면 auto-login redirect 가 stale 포트로 튀었음. 이번 회차에 잡고 worktree `.env.local` 만 수정.
- **Defense-in-depth 가 실제로 한 단계를 잡았다 (D8)** — 단순한 "보안 가드 있음" 가 아니라 "LLM 이 그것을 출력하면 차단된다" 를 fuzz로 입증.

## 아쉬운 점

- **Build-team 스킬을 호출하지 않고 단독 실행**했다. 사용자 지시에 "Phase A → build-team" 가 명시되어 있었는데, 작업 성격(스크립트 1개+리포트)이 5명 spawn 비용 대비 가치가 낮다고 판단해 생략. 향후 동일 패턴(QA 자동화) 에서는 명시적으로 룰을 만들거나 사용자 confirm 필요.
- **UI Playwright 검증이 얕다.** 1 케이스(D1 script-tag prompt) 만 시각 검증; 토스트가 너무 빨리 사라져 캡처 못 함. 다음 회차에는 `toast.error(...)` 호출 즉시 스크린샷 또는 console-error stub 으로 잡기.
- **A4 (zero-width)** 는 직관적으로 잡혀야 했는데 trim() 동작 확인을 hardcode 으로 못 했다. 다음 회차에서는 expect 를 `http_4xx` 로 박아서 fail 시키는 회귀 테스트로 승격.

## 학습

- **Next.js worktree 격리 시 `.env.local` 의 NEXTAUTH_URL/PORT 가 항상 worktree 별로 분기되어야 한다.** 안 그러면 auth callback 이 stale dev server 로 redirect — 다른 작업(TM-43 이 3043 사용 등) 충돌. → tech-note 후보.
- **LLM 의 거부 응답이 "valid JSON 이 아닌 prose"** 라는 것이 흔한 실패 모드라는 걸 정량 확인 (D1-D7 모두 동일 메시지). UI 메시지 카피라이팅을 LLM 응답 패턴에 맞춰 갱신 필요 (TM-59).
- **OpenAI 모델은 `<script>alert("xss")</script>` 같이 직설적인 XSS 요청은 거의 100% 거부**. 우회 시도는 D8 (Worker) 처럼 "원래 합법적인 API" 에 우회를 끼워넣는 형태로만 빠져나옴 → sandbox 가 잡음.
