# TM-102 r2 — TM-111 fix 라이브 검증

## TL;DR

[[2026-04-27-TM-111-fix|TM-111]] (PR #109 — `sanitizeForbiddenTokens` + multi-step → single-shot
fallback) 머지 후, [[2026-04-27-TM-108-quality-benchmark|TM-108]] 에서 `AI_MULTI_STEP=1 + gpt-4o`
조합으로 4/5 가 500 을 떴던 동일 5 케이스를 라이브 dev (port 3102) 에 재실행했다.

- **pass rate: 5/5 (100%)** — 이전 회 1/5 (20%) 대비 +80pp.
- **single-shot fallback 발동: 0/5** — sanitize 단계만으로 모든 sandbox 위반 패턴이 정제됐다.
- **sanitize 활성화: 1 회** (case 2 long-video, `scene-code[1]` — `const … = require(...)` 줄 제거).
- 평균 generate latency 22.9s (17.2 ~ 28.8s) — multi-step 4 scenes 기준 정상 범위.
- **결론**: TM-111 의 sanitize 가 가설(80% 이상 회복)을 초과해 100% 회복을 달성. 단일-샷 fallback 은
  보험으로 남고, 본 5 케이스에서는 gpt-4o 출력 위생 차이가 6 종 정규식 패턴으로 전부 흡수됨.

## 왜 / 배경

[[2026-04-27-TM-108-quality-benchmark|TM-108]] 의 핵심 발견은 multi-step 가용성 붕괴. RCA 는
"gpt-4o 가 scene-code 단계에서 `require(...)`, `globalThis.*`, transpile-깨는 토큰을 흘리고, sandbox
는 정상 차단하지만 composer/post 단계가 흡수하지 못한다" 였다. TM-111 은 두 단계 보호:

1. **sanitize 패스** (`sanitizeForbiddenTokens`) — validate 직전에 6 패턴 정제.
2. **single-shot fallback** — sanitize 실패 시 `generateAsset` 으로 dynamic import.

본 r2 는 후자가 작동하는지보다 **전자가 충분한지** 를 확인하는 것이 목적이었다. 결과: 본 5 케이스에서는
sanitize 만으로 전부 회복. fallback 은 미발동.

## 영향

### 가용성

- gpt-4o + multi-step 실패율 **80% → 0%** (n=5).
- TM-108 의 "default-off 권고" 가 부분 해소됨. 다만 5 케이스만으로 default-on 결정은 이르며, 더 큰
  표본/다양한 카테고리에서의 후속 검증 필요 (아래 "다음").

### 비용 / 시간

- 본 검증 비용: 5 generate 호출 (multi-step) ≈ **\$0.5 OpenAI** (judge 호출 없음).
- 평균 latency 22.9s — TM-108 의 baseline 10.7s 대비 ~2× (이전 보고서의 multi-step 22.5s 와 일치).

### 코드

- 검증 드라이버 신규 1 파일 (`tm-102-r2-verify.ts`, 175 lines). 기존 `pipeline.ts` / `generate.ts`
  변경 없음 — 본 r2 는 **검증 전용** 산출물이다.

## 한계 / 후속

- 표본 5 케이스 — 새 변종 (encoded require, `eval`, `Reflect.get`, `Proxy`) 노출 없음. 프로덕션
  telemetry 로 sanitize miss 패턴 모니터링 필요.
- gpt-4o-mini / gpt-4o-2024 등 다른 OpenAI 모델은 미검증.
- judge 점수는 본 r2 에서 측정하지 않음 — 가용성 게이트 통과 후 별도 trigger 로 TM-108 형식의
  품질 비교 가능.
- [ ] 후속 (선택): TM-108 형 judge 재실행으로 baseline vs full 의 품질 격차 측정 (\$1.4 OpenAI 추가).
