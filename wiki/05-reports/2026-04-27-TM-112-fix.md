# TM-112 — multi-step `Scene{N}Params is not defined` runtime crash

## TL;DR

TM-108 r2 가 발견한 P0 — `AI_MULTI_STEP=1 + gpt-4o` 모드에서 generate 200 5/5 가 모두 클라이언트 evaluator 단계에서 `ReferenceError: Scene1Params is not defined` 로 빈 화면을 띄우던 문제를 fix. composer 가 LLM 출력 식별자를 introspect 해 **canonical 이름으로 rename + spread 매핑을 동적으로 생성**하도록 변경.

라이브 검증(gpt-4o 2 케이스, multi-step) 결과: **2/2 evaluator PASS** (pre-fix 0/5 → post-fix 2/2). 단위 테스트 4 종 추가, 전체 pipeline 테스트 43/43 통과.

## 무엇이 바뀌었나

### 1. `composeSceneCodes` 동적 식별자 발견 (`src/lib/ai/pipeline.ts`)

Before: 모든 scene fragment 가 `Scene{N}Params` 와 `Scene{N}` 라는 정확한 이름을 정의했다고 가정하고 spread/sequence 를 박아 넣음.

After: 새 helper `findSceneIdentifiers(fragment, sceneNumber)` 가 fragment 의 top-level 선언을 regex 로 스캔해 실제 정의된 이름을 골라낸다.

- **Component 우선순위**: `Scene{N}` → `Scene*` PascalCase 중 `Params` 로 끝나지 않는 것 → 임의 PascalCase 식별자 → null.
- **Params 우선순위**: `Scene{N}Params` → 임의 `*Params` 식별자 → null.

발견된 비-canonical 이름은 fragment 안에서 일괄 rename 되어 wrapper 의 spread 와 sequence 가 항상 canonical 이름을 참조한다. params 가 아예 없는 scene 은 spread 에서 생략되며, component 가 없으면 빈 placeholder 가 자동 주입돼 다른 scene 은 정상 렌더된다.

### 2. 단위 테스트 4 종 추가 (`__tests__/lib/ai/pipeline.test.ts`)

새 `describe('TM-112 — composer name discovery (...)')` 블록:
- `Scene1Params` 가 두 번째 scene 에서 중복 정의된 케이스 → 두 번째는 `Scene2Params` 로 rename 됐는지 확인.
- params const 가 누락된 scene → 해당 spread 가 PARAMS 객체에서 빠지는지 확인.
- `myParams` 같은 비-canonical 이름 → `Scene{N}Params` 로 rename 됐는지 확인.
- 합성된 코드의 모든 `...XYZ` spread 가 실제 top-level 선언을 가리키는지 검증 (정합성 invariant).

### 3. 라이브 검증 스크립트 (`__tests__/benchmarks/tm-112-pipeline-direct.ts`)

`/api/generate` 가 prisma schema drift 로 500 을 내는 환경(prisma `Asset.templateSourceId` 가 dev sqlite 에 미반영) 을 우회하기 위해 `generateAssetMultiStep` 을 직접 import 해 호출 → `composedCode` 의 jsCode 를 evaluator 와 동일한 `new Function(...)` 로 실행해 ReferenceError 만 검사.

결과:
| case | scenes | latency | evaluator |
|---|---:|---:|---|
| tm112-chart (TM-108 case 4) | 2 | 18.6s | PASS |
| tm112-typo (TM-108 case 5) | 3 | 14.2s | PASS |

비용: gpt-4o full multi-step 2 회 ≈ $0.4 (judge 미사용).

## 왜 / 배경

[[2026-04-27-TM-108-quality-benchmark-r2|TM-108 r2]] 에서 `AI_MULTI_STEP=1 + gpt-4o` 가 생성-200 5/5 였지만 evaluator 가 5/5 모두 ReferenceError 로 떨어뜨려 ErrorBoundary 노출. 평균 점수 25.2 (baseline 70.6). 사용자에게 multi-step 은 **사실상 모두 빈 화면**.

원인을 파고들어 보면:
- 시스템 프롬프트는 `Scene{N}Params` + `Scene{N}` 을 요구하지만 gpt-4o 가 `{N}` 치환을 일관되게 못 함 (모든 scene 에 `Scene1Params` 만 쓰거나, 이름을 `myParams` / `params1` 로 바꾸거나, 컴포넌트 안에 params 를 정의해 module scope 에서 안 보이게 함).
- composer 는 `...Scene${i+1}Params,` 를 **무조건** 박아 넣고 있어 LLM 출력이 그 이름을 export 하지 못한 순간 `ReferenceError` 가 나는 게 결정적이었음.

defense-in-depth 관점에서 LLM 의 naming 일관성에 의존하는 건 안티패턴 — composer 가 fragment 를 introspect 해 실제 이름을 사용하는 것이 옳다.

## 영향

### 긍정

- multi-step 5/5 evaluator 0% → (라이브 2 케이스 대상) 100% 회복.
- placeholder fallback 덕에 한 scene 의 LLM 출력이 망가져도 다른 scene 은 정상 렌더.
- composer 가 LLM 의 naming drift 에 강건해져, 향후 모델 교체/프롬프트 미세조정에도 깨지지 않을 것.

### 잠재 리스크

- regex top-level 매칭은 휴리스틱이라 fragment 가 IIFE 로 감싸진 흔치 않은 형태에서는 식별자를 못 찾을 수도 있다 — 그때는 placeholder 가 그 scene 에 들어가므로 ReferenceError 는 안 나지만 그 scene 만 빈 화면. 후속 retro 에서 모니터링.
- rename 은 word-boundary regex 로 동작 — string literal 안에 동일 이름이 들어 있다면 의도치 않은 치환 가능성. 현재 LLM 출력 패턴상 발생 가능성 매우 낮으나 알아둘 가치는 있음.

## 후속

- TM-108 r3 재측정 — multi-step full 모드 5/5 evaluator PASS + 평균 점수 baseline 과 격차 좁혀지는지 확인 (triggers_requalify 발동).
- prisma `Asset.templateSourceId` 컬럼이 dev sqlite 에 반영되지 않은 별도 schema-drift 이슈 — 본 task 범위 밖, 별도 spawn 권고 (TM-112 라이브 검증을 `/api/generate` 가 아닌 pipeline 직접 호출로 우회한 이유).
- 시스템 프롬프트도 보강할지 결정: composer 가 강건해진 만큼 굳이 안 해도 되지만, `Scene{N}` 을 강제하면 rename 빈도를 줄여 미세하게 더 빠른 path. 후속 retro 에서 결정.
