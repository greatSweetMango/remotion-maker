# TM-67 회고 — transpileTSX 실패 retry-once

## 잘된 점

- **TM-51 패턴 재사용으로 design 비용 0**. `generateOnce` 헬퍼가 이미 soft-fail discriminated union 을 반환하는 형태로 분리되어 있었기에, `transpile_error` variant 를 추가하고 `generateAsset` 의 dispatch 분기만 늘리면 끝났다.
- **에러 메시지 인젝션이 핵심**. 단순 retry (동일 프롬프트 + temperature jitter) 는 통계적 효과만 기대 가능하지만, sucrase 의 실제 에러 메시지를 system prompt 에 박아넣으면 모델이 무엇을 고쳐야 할지 명시적으로 안다 — 비용 대비 효과 큰 신호.
- **테스트 isolation 확인**. placeholder→transpile, transpile→placeholder cross 케이스도 명시적으로 throw 분기. path 격리 테스트 (placeholder 1차 시 transpileTSX 1회만 호출) 로 회귀 방어선 확보.

## 아쉬운 점 / 개선

- **샘플이 2건뿐**. r2 에서 발견된 transpile fail 은 ig-02, tr-10 두 건. retry 가 정말 둘 다 self-heal 하는지는 r3 실측이 답. 만약 모델이 동일 syntax 오류를 반복하면 (e.g. JSX child 처리 약점이 모델 자체 문제) retry 만으론 부족 — sucrase 통과 후 evaluator 단까지 끝까지 가는 e2e validate-then-retry 까지 확장 필요할 수 있음.
- **prompt token 비용**. `buildTranspileRetryReinforcement` 가 ~600 token 추가. retry 시점에만 발동이라 평균 비용 영향은 작지만, 만약 transpile fail 비율이 높아지면 누적 비용 모니터링 필요. r3 에서 retry rate 측정.
- **temperature 변경은 안 함**. 같은 모델, 같은 temperature, 다른 system prompt 로만 retry. retry 가 또 transpile fail 시 다음 iter 에서 temperature jitter 추가 검토.
