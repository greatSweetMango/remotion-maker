# TM-36 회고 — ADR 번호 충돌 회피

## 요약

3-tier Ralph 루프에서 N개 TeamLead가 병렬로 ADR을 만들 때 같은 `NNNN`을 부여하는 race를 제거. Orchestrator가 commit 직전 단일 직렬화 지점에서 NNNN을 부여하는 placeholder 패턴으로 박제.

## 변경 (단일 PR, infra)

- `.claude/commands/orchestrate.md` Step 5에 **Step 5-pre** 절 추가: max_nnnn 스캔 → task_id 정렬 → 순차 부여 → token/path 일괄 치환 → ADR 인덱스 갱신. push 실패 시 rebase + 재산정 fallback.
- `prompts/team-lead.md` Phase F: ADR placeholder 형식 강제 (`PENDING-<task_id>-<slug>.md`, self-ref `ADR-PENDING-<task_id>`). NNNN 직접 부여 금지.
- `.claude/agents/pm.md` 컨텍스트 템플릿의 산출물 경로 컨벤션 갱신.
- `wiki/CLAUDE.md` §8 Architect 행 명세 갱신.
- ADR-0012 추가.

## 왜 placeholder인가

- timestamp 기반 임시 ID 대비 task_id가 PR/log와 1:1 매핑되어 디버깅이 명확.
- filesystem 락은 worktree가 분리되어 의미 없음.
- TeamLead가 main push를 한다는 대안은 wiki = main 단독 ownership 위반.
- 결정성: 같은 입력에 대해 항상 같은 NNNN 매핑 (task_id 정렬 키).

## 후속

- 실제 N≥2 병렬 iter 한 번 돌려서 알고리즘 검증 필요. (현재 환경에서는 dry-run 단계)
- 같은 패턴을 회고/QA 파일에도 적용할 필요 없음(파일명에 `<date>-<task_id>` 포함되어 자연스럽게 충돌 없음).
