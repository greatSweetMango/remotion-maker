# TM-109 — template monthlyUsage 중복 차감 server-side 가드

## TL;DR

- TM-106 retro follow-up #1: `/api/edit` 의 template path가 매 edit마다 `monthlyUsage +1` 했던 문제 server-side에서 차단.
- `Asset.templateSourceId` 컬럼 추가 + 라우트에서 `(userId, templateSourceId)` lookup으로 재머터리얼라이즈 가드.
- 동일 `template-…` 재전송 5회 → `monthlyUsage` 정확히 1번만 차감하는 단위 테스트 통과.

## 무엇이 바뀌었나

- `prisma/schema.prisma` — `Asset.templateSourceId String?` + `@@index([userId, templateSourceId])`.
- `src/app/api/edit/route.ts` — template branch에서 기존 머터리얼라이즈 row가 있으면 그 row id로 pivot하고 `asset.create` / `monthlyUsage +1` 모두 skip. reservation/refund/version write 모두 effective id 기준.
- `__tests__/api/edit/template-quota.test.ts` — 신규 (2 cases): 5회 sequential template edit → `monthlyUsage === 1`, first materialize → `templateSourceId` 스탬핑 검증.

## 왜 / 배경

TM-106 retro에서 "현재 `/api/edit` 가 모든 template edit에서 `monthlyUsage +1` 차감 — 1회 머터리얼라이즈에만 차감해야 의미적으로 맞음"으로 후속 권고됨. TM-106 클라이언트 fix(`ADD_VERSION` action에 `newAssetId` 채택)로 happy path는 정상이지만, 클라이언트 버그/공격적 사용자가 `assetId: "template-…"` 를 계속 보내면 매번 새 Asset row가 생기고 `monthlyUsage` 가 무한 차감되는 결함이 남아있었다. defence-in-depth로 server에서 가드.

## 영향

- **코드/시스템**: `/api/edit` template path만 변경. 기존 cuid asset path는 영향 없음. TM-93 `reserveEditSlot` atomic UPDATE도 변경 없음.
- **사용자/제품**: FREE tier 월 quota 보호. 동일 template 5회 편집 시 1슬롯만 차감.
- **비용/성능**: TM-109 가드 query 1회 추가 (`asset.findFirst` on `templateSourceId` index). `Promise.all` 안에 들어있어 latency 영향 minimal.
- **DB 마이그레이션**: `prisma db push` 로 nullable 컬럼 + 인덱스 추가. 기존 row는 모두 `templateSourceId = NULL` 로 시작 — 과거에 만들어진 row에 대한 backfill은 불필요 (어차피 `template-…` 재전송이 와도 NULL 매치는 안 되므로 새 row 생성 → 그 시점부터 가드 활성화).
