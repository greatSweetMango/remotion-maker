# TM-110 — ParticlePhysics SSR hydration mismatch fix

## TL;DR
QA r2 (TM-84) reported a React hydration warning whenever
`src/remotion/templates/ParticlePhysics.tsx` was rendered inside a
`<Player>`. QA's hypothesis was `Math.random()` in render. **That hypothesis
was wrong** — the template already uses a deterministic `Math.sin`-based
seeded RNG (since TM-38). The real cause was floating-point pixel
serialization: server-rendered CSS strings rounded to 3 decimals (`left:
"801.427px"`) while the client React renderer compared against the raw JS
number (`801.4273423103696`), which React 19 flags as a hydration
attribute mismatch.

Fix: round the per-particle `x / y / size / alpha` and the derived
`boxShadow` blur radius to 3 decimals before passing them to inline
`style`. Server output and client value now stringify to the same digits,
hydration is byte-stable.

## Repro
- Created a temp dev probe at `/dev-particle-hydration` that renders
  `ParticlePhysics` inside `@remotion/player` `<Player>` (1920×1080, autoPlay).
- Loaded over Playwright; console captured one `error`-level "A tree
  hydrated but some attributes of the server rendered HTML didn't match
  the client properties" with a diff showing pixel values like
  `+left: 801.4273423103696 / -left: "801.427px"` for every particle div.
- Probe was removed before commit; the fix is verified in-place by reloading
  the same probe and observing 0 hydration warnings (only unrelated
  NextAuth 500 + Remotion-license / AudioContext warnings remain on the
  page, which are environmental noise from running without a DB).

## Why QA's "Math.random" hypothesis was wrong
- Code search: `grep -n "Math.random" src/remotion/templates/` shows only
  `BezierPath.tsx` (in a comment explaining why it doesn't use it).
  `ParticlePhysics.tsx` has no `Math.random()` call at any point in its
  history.
- The seeded RNG `(s = Math.sin(seed + k) * 43758.5453; s - Math.floor(s))`
  is bit-for-bit identical on Node.js (server) and V8 (client). Both
  hydration sides compute the **same numbers**.
- The mismatch came from React's CSS-attribute reconciliation, not from
  the simulation. React on the server stringifies a numeric `style.left`
  to a 3-decimal `"<n>px"` value; on the client the comparison is against
  the raw JS number, and 16-digit floats don't equal their truncated
  string form.

## Fix
`src/remotion/templates/ParticlePhysics.tsx`:

```ts
const r3 = (n: number) => Math.round(n * 1000) / 1000;
return {
  x: r3(x),
  y: r3(y),
  size: r3(size),
  color: color as string,
  alpha: r3(0.55 + rnd(8) * 0.4),
};
```

Plus `boxShadow: `0 0 ${Math.round(p.size * 1.6 * 1000) / 1000}px ${p.color}``
to keep the same precision contract.

No behavioral change — 3-decimal pixel precision is well below
sub-pixel rendering granularity.
