# TM-92 회고

## 잘 된 점

1. **TM-83 의 발견 → tech-note → fix 까지 1 hop**. 회고/노트에 적힌 fix sketch 가
   거의 그대로 작동했다. tech-note 의 가치 입증.
2. **단일 statement atomic 패턴 채택** — SQLite advisory lock 이나 Prisma
   `$transaction({ isolationLevel: ... })` 같은 무거운 도구 없이 `updateMany`
   조건부 업데이트로 race 종결. 코드 변경 표면적 최소.
3. **회귀 테스트가 race 자체를 검증** — mocked prisma 의 `updateMany` 를 mutex
   직렬화로 모델링해 SQLite "single-statement atomic" 의미를 그대로 재현. 단순
   "limit 도달 시 429" 단위 검사가 아니라 **동시성 분포** 를 assert.

## 아쉬운 점

1. **mock 기반 회귀 테스트의 한계** — 진짜 SQLite + Prisma 와의 race 는 별도
   integration 테스트가 있어야 100% 안전. 현재는 jest mock 가 atomic 의미를
   "흉내" 낸 수준. 실DB perf 테스트 (TM-83 의 `tm-83-concurrent.mjs`) 를 CI 에
   엮는 follow-up 가치 있음.
2. **환불 경로가 늘면서 코드 분기 증가** — clarify, refusal, 5xx 세 곳 모두
   `decrement: 1`. 누락 시 무료 quota 가 새는 방향이라 보안상은 안전하지만, 새
   에러 경로 추가 시 잊지 않게 헬퍼 (`refundQuotaSlot(userId)`) 로 추출 검토.
3. **PRO 200 동시 호출 시나리오 미검증** — FREE limit=3 만 직접 검증. PRO 200
   에서도 동일 패턴이 동작하는 건 코드상 자명하지만 명시적 테스트 없음.

## 다음에 시도

1. **헬퍼 추출**: `reserveGenerationSlot(userId, tier)` / `refundGenerationSlot(userId)`
   를 `src/lib/usage.ts` 에 두고 route 는 호출만. 환불 누락 방지.
2. **integration race 테스트**: `tm-83-concurrent.mjs` 를 jest 의 SQLite 격리
   DB 와 결합해 CI 에서 매 PR 마다 race 회귀 잡기.
3. **다른 quota 게이트 점검**: edit / upload 한도도 같은 TOCTOU 가 있는지 audit.
   특히 `editUsage` 카운터는 JSON column 이라 patch 가 read-modify-write — 더
   위험할 가능성.
