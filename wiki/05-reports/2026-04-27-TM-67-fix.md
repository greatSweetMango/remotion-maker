# TM-67 — transpileTSX 실패 retry-once fix

## TL;DR

- TM-41 r2 QA 에서 발견된 transpileTSX(sucrase) 실패 2건 (ig-02 JSX child, tr-10 stray semicolon — gpt-4o-mini)을 retry-once 로 자가 치유.
- TM-51 의 `generateOnce` placeholder retry 패턴을 동일하게 transpile 실패 케이스에 적용 — 1차 실패 시 sucrase 에러 메시지를 시스템 프롬프트에 주입한 syntax-correctness 강화 프롬프트로 1회 재시도. 2회 모두 실패 → user-facing error.
- 신규 의존성 0, 신규 테스트 3건 추가 (총 311 pass / 2 skip — 회귀 0).

## 문제

TM-41 r2 QA (`wiki/05-reports/2026-04-27-TM-41-qa-r2.md`):

| ID | tier | category | error |
|---|---|---|---|
| ig-02 | FREE | infographic | `Unexpected token when processing JSX children` |
| tr-10 | FREE | transition  | `Unexpected token, expected ";" (5:89)` |

응답 자체는 JSON 으로 파싱 성공, sandbox `validateCode` 통과, placeholder 도 아님 — 그러나 `transpileTSX` 단계에서 sucrase 가 syntax error 를 던짐. 이전엔 그 에러가 그대로 위로 propagate 되어 사용자에게 `Failed to generate` 같은 모호한 메시지로 표면화. 모델 출력의 단발성 syntax 오류는 retry 로 거의 항상 자가 치유 가능한 케이스인데도 1-shot 폴리시 때문에 fail.

## 원인 분석

1. **`generateOnce` 의 transpile 단계가 hard-throw**: `await transpileTSX(sanitized)` 가 try 없이 호출. sucrase `SyntaxError` 가 위로 그대로 올라가 `generateAsset` 의 caller (route handler) 까지 unhandled.
2. **TM-51 retry 는 placeholder 만 커버**: `detectPlaceholderCode` 통과 케이스만 retry. transpile 실패는 retry 패스 밖.
3. **모델이 자기 syntax 오류를 인지 못함**: 1차 응답 + 시스템 프롬프트만으로 다시 호출하면 동일 prompt → 동일 hash → 캐시 히트는 아니지만 통계적으로 비슷한 출력. 에러 메시지를 `next attempt` 의 시스템 프롬프트에 주입해야 모델이 무엇을 고쳐야 할지 안다.

## 조치

### 1. `generateOnce` 가 transpile 실패를 soft-fail 로 반환 (`src/lib/ai/generate.ts`)

```ts
let jsCode: string;
try {
  jsCode = await transpileTSX(sanitized);
} catch (err) {
  const errorMessage = err instanceof Error ? err.message : String(err);
  return { kind: 'transpile_error', rawCode: code, errorMessage };
}
```

return type 에 `| { kind: 'transpile_error'; rawCode; errorMessage }` 추가. caller 가 retry 결정 가능.

### 2. `generateAsset` retry-on-transpile (`src/lib/ai/generate.ts`)

기존 placeholder retry path 와 병렬로 transpile_error path 신설. 동일 `generateOnce` 헬퍼 재사용:

- 1차 transpile_error → `GENERATION_WITH_CLARIFY_SYSTEM_PROMPT + buildTranspileRetryReinforcement(errorMessage)` 로 2차 호출
- 2차 response → 그대로 반환
- 2차 transpile_error → user-facing throw `"AI produced TSX that failed to transpile twice (last error: ...)"`
- 2차 placeholder (드물지만 가능) → user-facing throw

placeholder→transpile 또는 transpile→placeholder cross 케이스도 명시적으로 처리 (3가지 throw 메시지 분리).

### 3. syntax-correctness 강화 프롬프트 (`src/lib/ai/prompts.ts`)

신규 export `buildTranspileRetryReinforcement(transpileErrorMessage: string)` — 함수형 (에러 메시지 interpolation).

핵심 문구:
- 실제 sucrase 에러 메시지 그대로 주입 → 모델이 어느 줄/토큰을 고쳐야 할지 안다.
- 5가지 syntax rule (JSX 균형, 중괄호 내 expression vs statement, 세미콜론, raw `<`/`>` escape, 표준 TSX only)
- "Mentally lint the code character-by-character" — 모델 self-check 유도.

첫 호출은 기존 prompt 그대로 — token 사용 증가는 retry 시점에만 (TM-51 패턴 동일).

### 4. 테스트 (`__tests__/lib/ai/generate.test.ts`)

신규 `describe('generateAsset retry on transpile failure (TM-67)')` 3 케이스:

- **retry succeeds**: `transpileTSX` 1차 throw → 2차 pass. 2차 호출의 system prompt 에 `SYNTAX VALIDITY ENFORCEMENT` + 원본 에러 메시지 포함 검증.
- **2회 모두 throw**: 두번째 transpile 도 실패 → `/transpile|failed to transpile/i` matcher.
- **path isolation**: placeholder 1차 → response 2차 시 TM-51 path 만 발동, transpileTSX 는 1회 호출 (placeholder 응답에는 transpile 안 됨), system prompt 에 `ANTI-PLACEHOLDER ENFORCEMENT` 만 포함, `SYNTAX VALIDITY ENFORCEMENT` 부재 확인.

기존 12 테스트 무회귀.
