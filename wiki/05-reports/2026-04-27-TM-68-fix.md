# TM-68 — entity-count clarify gate

## TL;DR

- TM-41 r2 QA에서 dv-05 / ig-03 두 케이스가 clarify 오트리거로 fail.
- TM-52 의 `scoreConcreteness` 가 두 prompt 다 `isConcrete=true` 로 분류해 forced-generate retry 가 발화 — 그런데도 LLM 이 retry 에서까지 `mode=clarify` 를 유지해 그대로 surfaced.
- TM-68 은 그 위에 **entity-count gate** 를 얹는다. 사용자가 명시적으로 "8개 막대", "3가지 특징", "5 bars" 처럼 갯수를 셌다면 → 한 번 더 hardened retry (TM-68 reinforcement: 갯수를 그대로 인용) 로 강제 generate.
- 31 unit tests + 3 e2e tests 추가, 전체 347 pass.

## 변경 요약

| 항목 | 내용 |
|---|---|
| 신규 export | `extractEntityCount(prompt) → number`, `ENTITY_COUNT_SKIP_THRESHOLD = 2`, `buildEntityCountReinforcement(n)` |
| `ConcretenessReport` 확장 | `entityCount: number`, `forceSkipClarify: boolean` |
| `forceSkipClarify` 트리거 | `entityCount ≥ 2` OR `hits.length ≥ 3` |
| `generate.ts` | TM-52 forced retry 가 `clarify` 를 또 반환하면, `forceSkipClarify && entityCount > 0` 일 때 한 번 더 hardened retry. 갯수가 0 (예: 시간 단위만 있는 ig-03 케이스) 이면 인용할 숫자가 없으므로 스킵. |

### 휴리스틱 디테일

`extractEntityCount` 가 의도적으로 **제외**하는 카운터:
- 시간 단위: `초/s/seconds/fps/frames/ms` — 길이 정보지 entity 갯수가 아님 (ig-03 의 "30s vs 5s" 가 8 로 잘못 잡히는 것을 막음)

대신 **포함**하는 카운터:
- KO: `가지/개/단계/단/회/번/컷/항목/요소/줄/행/열/칸/장/쪽/페이지`
- EN: `items/elements/bars/dots/circles/squares/stars/lines/columns/rows/sections/panels/tiles/steps/points/nodes/cards/icons` (plural)
- Grid: `NxM` → 곱 (예: `3x3` → 9)

## 영향 범위

- `src/lib/ai/clarify-gate.ts` — entity-count 로직 + 새 reinforcement
- `src/lib/ai/generate.ts` — TM-52 forced retry 다음에 TM-68 hardened retry 분기 추가
- `__tests__/lib/ai/clarify-gate.test.ts` — 31 → 32 unit + 3 새 e2e

기존 TM-52 테스트는 그대로 통과 (entity-count 가 0 인 vague prompt 는 새 코드패스 미발화).

| dv-05 | "...빨강/초록 캔들, **8개 막대**" | 8 | ✓ | TM-52 forced retry 도 clarify면 → TM-68 hardened retry (갯수 8 인용) |
