## What worked

- **누적 fix 효과 검증**: TM-66/67/68/69 r2 fix 가 r3 에서 모두 회복 확인 — transpile 0, clarify 0, JSON parse 0.
- **r2→r3 8.3pp 개선** (91.7% → 100%): 한 회차에 acceptance 도달. variance 가능성 있으나 60 runs 0 fail 은 통계적으로 강한 신호.
- **smoke-then-full 전략**: 10 runs (~34s) smoke 로 harness 동작 사전 확인 후 60 runs 풀 런. r3 전 시도 stall 회피.
- **harness 변경 없음**: r1/r2 동일 코드 재사용 — 측정 일관성 유지.
- **PRO 100% 유지** (r2 부터): TM-51 retry-on-stub 안정화.

## What didn't

- **전체 응답 p50 +2% 변동** (7,854 → 8,045 ms): acceptance 외 지표지만 약한 회귀. PRO sample = 10 으로 작아 노이즈 가능성.
- **TTFB p95 변동** (646 → 2,817 ms): n=10 sample variance. 더 큰 sample 또는 정기 측정으로 추적 권장.

## Lessons

- **r3 의 가치**: r2 의 "1.7pp 개선" 처럼 조금씩 수렴하는 게 아니라, fix task 4개 누적이 한 번에 100% 도달시킴 — fix-induced regression 이 아니라 진짜 fix 였음.
- **acceptance 4 axis 정량화의 가치**: pass_rate 만 보면 r2 대비 +8.3pp 단순 측정이지만, evaluator 실패/콘솔 에러/TTFB 별도 측정 → fix 효과 분리 검증 가능.
- **smoke 부분 측정의 비용 효과**: 10 runs $0.05, 60 runs $0.3 — smoke 가 hung/wrong-config 를 일찍 잡으면 풀 런 폐기 비용 절감.
