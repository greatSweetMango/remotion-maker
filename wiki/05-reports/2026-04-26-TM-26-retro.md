  - `scripts/launchd/README.md` — 등록/해제 절차

## 잘된 점

- **TeamLead 단독 라우팅 유지**: 본 task 는 신규 텍스트 파일 6건 생성 + 사양 명확. build-team 5명 spawn 비용 회피. 메모리 `feedback_automation_preference.md` 정신 일치. 대신 그 분기 기준이 prompts/team-lead.md 에 명시 안 된 점은 다음 주기 개선안 (W17 §5 제안 4) 으로 박제.
- **Dry-run fallback 설계**: `claude` CLI 부재 환경에서도 schema-only stub 생성 → CI/테스트 시나리오에서 스크립트 자체 검증 가능. TM-19 의 graceful-degrade 패턴 (jq 부재 시 시스템 흐름 차단 X) 재사용.
- **launchd plist 의 month-end 트릭**: launchd 가 "마지막 날" 직접 표현 없으므로 28-31일 모두 트리거하되 셸에서 `tomorrow_month != today_month` 검사. macOS BSD date 와 GNU date 양쪽 호환.
- **wiki 단독 소유 정책 준수**: weekly 리포트는 worktree 가 아닌 main 에서 commit 되어야 정책 일관 — 스크립트가 `git checkout main && git pull --ff-only` 를 launchd 플로우에 강제.
- **첫 시드를 stub 이 아닌 실데이터 기반**: 5개 micro retro 를 손으로 분석한 결과로 §5 제안 6건 도출. 다음 주기 자동 결과와 비교 가능한 baseline 확보.

## 아쉬운 점 / 리스크

- **claude CLI 헤드리스 호출 인자**: `claude -p --append-system-prompt` 형식이 실제 EasyMake 환경의 claude CLI 와 100% 호환되는지 미검증 (라이브 호출 안 함). 다음 주기 첫 자동 실행 시 검증 필요 — 실패 시 fallback dry-run 으로 빠지므로 폭주는 없음.
- **launchctl 자동 등록 X**: 사용자가 수동으로 `launchctl load` 해야 함. plist 의 `__REPO_ROOT__` 치환도 README 절차 따라 사람 작업. 잊으면 다음 주기 자동 실행 안 됨 → W17 §5 제안 5 로 박제.
- **데이터 부족 (n=5)**: 첫 weekly 의 §5 제안은 통계 기반 X, 정성 관찰 기반. 다음 2-3주기 후 trend 가 잡혀야 의미있는 SOP 갱신 가능.
- **build-team 미호출**: prompts/team-lead.md 가 build-team 호출을 디폴트로 명시하지만 본 task 는 단독 처리. SOP-실태 불일치는 W17 §5 제안 4 에서 다룸.

## 다음에 가져갈 결정

- **인프라 task 표준 흐름**: 신규 prompt + 스크립트 + plist + dry-run 시드. TM-26 을 템플릿으로 향후 cron 기반 agent 추가 시 재사용.
- **Macro 리포트 시계열 시작점**: 2026-W17 = baseline. 자동 결과와 시드를 W18 에서 비교 — diff 가 크면 prompts/meta-analyzer.md 튜닝 트리거.
- **점진 자율화 게이트**: 사람 검토 0건 = 신호로 받음. blueprint §11 의 다음 단계 진입 시 "주간 사람 검토 1회 의무" 추가 검토 (W17 §6 리스크 4).

## SOP 갱신 권장

- `prompts/team-lead.md` 에 분기 기준 명시: "신규 코드/wiki 파일 ≤ 6건 + complexity ≤ medium + 외부 리서치 불필요 → TeamLead 단독, 그 외 → build-team" (W17 §5 제안 4 와 직결).
- 새 cron/launchd agent 추가 시 체크리스트:
  1. 진입점 스크립트는 dry-run fallback 보유
  2. plist 의 절대경로는 `__REPO_ROOT__` placeholder + README sed 치환

  4. 분석/실행 시간 상한 (timeout) 설정
  5. main 브랜치 강제 (wiki 단독 소유 정책)
  6. 산출물 스키마 자동 검증 (필수 섹션 grep)
