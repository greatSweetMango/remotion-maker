# TM-93 — /api/edit editUsage TOCTOU 픽스

TM-92 retro 의 follow-up §2.3 에서 명시적으로 의심되었던
`editUsage` JSON column 의 read-modify-write race 를 픽스한다.

## 발견 경로

- TM-92 회고: `editUsage` 카운터는 JSON column 이라 patch 가
  `JSON.parse → 객체 spread → JSON.stringify` 의 3-step. /api/generate
  와 똑같이 (혹은 더) 위험할 가능성 — audit 권고.
- 실제 코드 (`src/app/api/edit/route.ts` 38-55 라인) 검토 결과 동일 race
  pattern 확인:
  ```
  findUnique → JSON.parse(editUsage) → editUsage[key] || 0 → checkEditLimit
  → editAsset (LLM, 수 초) → user.update({ editUsage: stringify({...spread, [key]: count+1}) })
  ```
  N 개 동시 요청이 같은 `editCount` 를 읽고 모두 통과해 모두 +1 → 최종
  카운터는 `count + N` 이지만 LLM 호출은 N 회 (FREE editsPerAsset=3 인데
  10 회 호출 가능).

## 왜 TM-92 의 `updateMany` 패턴을 그대로 못 쓰나

`monthlyUsage` 는 `Int` 컬럼이라
`prisma.user.updateMany({ where: { monthlyUsage: { lt: limit } }, data: { increment: 1 } })`
한 줄로 끝났다. `editUsage` 는 SQLite `String`(JSON) 으로 직렬화돼 있고,
키가 `assetId` 동적이라 Prisma 의 type-safe `where` 로는 per-key
조건부 비교가 불가능하다.

대안:
1. **schema 분리** — 새 `AssetEditCount` 테이블. atomic `updateMany` 자명.
   하지만 **DB migration 필요** → escalate 정책 트리거.
2. **`$transaction` + isolation level** — SQLite 는 진정한 row lock 이
   없고, Prisma SQLite adapter 는 `Serializable` 하나뿐. statement 직렬화
   비용 큼.
3. **raw `UPDATE` with SQLite JSON1** — `json_extract` / `json_set` 을
   single statement 안에서 결합. SQLite 는 한 statement 를 atomic 으로
   실행 → TM-92 의 `updateMany` 와 동일 의미.

(3) 채택. migration 없음, 코드 표면 ~30 라인.

## 픽스

### `src/lib/usage.ts` — 신규 헬퍼

```ts
export async function reserveEditSlot({ userId, assetId, limit }) {
  if (!isValidAssetIdForUsageKey(assetId)) {
    throw new Error('reserveEditSlot: invalid assetId');
  }
  const path = `$."${assetId}"`;
  const effectiveLimit = Number.isFinite(limit) ? limit : Number.MAX_SAFE_INTEGER;
  const affected = await prisma.$executeRaw`
    UPDATE User
    SET editUsage = json_set(
      editUsage,
      ${path},
      COALESCE(json_extract(editUsage, ${path}), 0) + 1
    )
    WHERE id = ${userId}
      AND COALESCE(json_extract(editUsage, ${path}), 0) < ${effectiveLimit}
  `;
  return affected === 1;
}

export async function refundEditSlot({ userId, assetId }) { /* MAX(... - 1, 0) */ }
```

핵심 invariant:
- SQLite single-statement atomicity → 동시 N 개 호출이 와도 정확히
  `limit - count` 개만 `affected = 1` 을 받는다. 나머지는 `0`.
- `assetId` 는 JSON path 키로 직접 interpolate 된다 (Prisma tagged
  template 의 식별자 자리는 parameter binding 안 됨). 따라서 input
  whitelist (`/^[a-zA-Z0-9_-]+$/`, len ≤ 128) 로 SQL/JSON-path
  injection 차단. cuids + `template-<cuid>` 모두 통과.
- `limit` 은 ${} parameter binding (안전).
- `Infinity` (PRO `editsPerAsset`) → `Number.MAX_SAFE_INTEGER` 로 변환,
  WHERE 사실상 무제한.

### `src/app/api/edit/route.ts`

LLM 호출 **이전** 에 `reserveEditSlot` 호출. false 면 즉시 429.
catch 블록에서 `refundEditSlot` 호출 (5xx 가 quota 를 영구 소비하지
않게). 기존 `prisma.user.update({ editUsage: stringify(...) })` 두 곳
(template 분기 + 일반 분기) 제거 — 이미 reserve 에서 atomic 하게 +1.
template 분기의 `monthlyUsage: { increment: 1 }` 는 그대로 유지 (새
asset 생성 시 monthlyUsage 도 차감).

## 회귀 테스트 — `__tests__/api/edit/quota-toctou.test.ts`

3 개 케이스:
1. **race 직접 검증** — 10 동시 호출, FREE limit=3, mocked `$executeRaw`
   가 mutex 로 single-statement 직렬화 모델. 결과: `200 × 3`,
   `429 × 7`, `editAsset` 호출 3 회.
2. **refund** — `editAsset` throw 시 reserve 했던 slot 이 환불돼
   카운터가 원상복구.
3. **assetId 화이트리스트** — `"abc'\"]injection"` 같은 입력은 raw SQL
   에 닿기 전에 400 으로 거절.

전체 `__tests__/api/` + `__tests__/lib/usage.test.ts`: 77 passed,
0 regressions.

## 잔여 과제 / 후속

- 진짜 SQLite + Prisma 와의 race 는 별도 integration 테스트로 검증해야
  100% 안전. 현재는 mocked $executeRaw 가 atomic 의미를 mutex 로 흉내.
  TM-92 retro 와 같은 후속 (실DB integration suite + CI 연결) 권고.
- `reserveEditSlot` / `refundEditSlot` 시그니처는 `usage.ts` 로 통일.
  TM-92 가 inline 에 둔 `monthlyUsage` reserve/refund 도 추후 같은
  파일로 추출 검토 (별 task).
