# TM-92 fix — /api/generate quota TOCTOU close

## 한 줄

`findUnique → in-memory check → generateAsset → update` 시퀀스를 **`updateMany` (조건부) → generateAsset → 환불** 패턴으로 교체해 동시성 race 를 닫았다.

## 변경 위치

- `src/app/api/generate/route.ts`
- 새 테스트: `__tests__/api/generate/quota-toctou.test.ts`
- 보강: `__tests__/api/generate/length-cap.test.ts` (mock 에 `updateMany` 추가)

## 동작

```ts
const limit = TIER_LIMITS[user.tier].monthlyGenerations;
const reserved = await prisma.user.updateMany({
  where: { id: user.id, monthlyUsage: { lt: limit } },
  data: { monthlyUsage: { increment: 1 } },
});
if (reserved.count === 0) return 429;

try {
  const result = await generateAsset(...);
  if (result.type === 'clarify') {
    // 환불: clarify 는 과금 X
    await prisma.user.update({ where: { id: user.id }, data: { monthlyUsage: { decrement: 1 } } });
    return ...;
  }
  // 생성 성공: 이미 reserve 했으니 추가 increment 없음
  return ...;
} catch (e) {
  // refusal / 5xx 모두 환불
  await prisma.user.update({ where: { id: user.id }, data: { monthlyUsage: { decrement: 1 } } });
  throw;
}
```

## 왜 atomic 인가

- SQLite 는 단일 SQL 문 (`UPDATE ... WHERE ... AND monthlyUsage < N`) 을 한 트랜잭션으로 처리. Prisma `updateMany` 는 그대로 한 문장으로 내려간다.
- N 개 동시 요청이 같은 row 의 `monthlyUsage=0` 에서 출발해도, DB 쪽에서 직렬화된 비교+증가가 일어나 정확히 `limit - usage` 개만 `count: 1` 을 받는다. 나머지는 `count: 0` → 429.
- 별도 트랜잭션/락 불필요. SQLite 에서 advisory lock 도구가 빈약한 점을 우회.

## 환불 경로

| 경로 | 상태 | 환불? |
|---|---|---|
| 생성 성공 | 200 generate | X (이미 reserve 됨) |
| Clarify-only | 200 clarify | O |
| `AiRefusalError` | 400 | O |
| 기타 throw | 500 | O |

## 회귀 테스트

`__tests__/api/generate/quota-toctou.test.ts`:

- 10 동시 FREE 요청 (limit=3) → status 분포 `200 × 3 / 429 × 7`. `monthlyUsage === 3`. `generateAsset` 정확히 3회 호출.
- Clarify 환불 시 카운터 0 유지.

mock 한 `updateMany` 는 한 번에 하나씩 비교+증가하도록 mutex 직렬화. 이는 SQLite "단일 statement = atomic" 의 모델.

## 남은 위험

- Prisma 5.x → 6.x 업그레이드 시 `updateMany` 의 `where` 에 scalar filter (`monthlyUsage: { lt }`) 가 계속 허용되는지 회귀 검증 필요. (현 v6 에서 동작 확인.)
- Postgres 등 다른 RDBMS 로 옮기면 **default isolation level** 이 바뀌어 race 모양이 달라질 수 있다. 그래도 같은 conditional update 패턴은 안전.
